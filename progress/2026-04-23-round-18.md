## Scenario

Two-agent swarm: `Alice` (math, tool `percent_of`) + `Bob` (research, tool `research_fact` hard-coded to return "$6.75T FY2024"). Question: "What is 15% of the US federal government's total spending in fiscal year 2024?"

## Finding #1 — Silent multi-LLM-per-node token undercount (BUG — fixed)

### Discovery

The trace *looked* fine. But `inspect_trace.py` showed Bob's `state_after` contained **4** new messages (2 AI + 2 tool), meaning Bob had made **two** LLM calls inside a single super-step — once to decide to call `research_fact`, once to decide to hand off back to Alice. Yet Chronos had only recorded the usage of the **last** `AIMessage`, missing ~50% of Bob's actual token consumption.

### Root cause

The three LangGraph usage extractors (`aimessage_usage_extractor`,
`anthropic_usage_extractor`, `openai_usage_extractor`) all used "last
message wins" semantics. They were correct for every graph Chronos had
seen through R17 because each LangGraph node there happened to emit
exactly one `AIMessage` per super-step. Swarm's `create_react_agent`-as-node
pattern breaks that — the inner ReAct loop runs to completion (N LLM calls)
inside what the parent graph sees as a single step.

### Bug severity

- Silent (numbers look valid)
- Systematic undercount, up to ~50% on swarm-style graphs
- Present in all three extractors (Anthropic / OpenAI / generic AIMessage)
- Latent in R17 supervisor dogfood — not triggered then, but on re-run with the fix, `research_expert` node now reports `1957+283` vs pre-fix `1755+271` (≈10% previously-invisible undercount)

### Fix

ADR-012. Added `_new_messages(ctx)` helper that diffs `post_values["messages"]`
against `pre_values["messages"]` (both already plumbed since R15/ADR-011 but
never used). All three extractors now sum usage across **all** new
`AIMessage` objects, not just the last.

```python
def _new_messages(ctx: UsageContext) -> list[Any]:
    pre = (ctx.pre_values or {}).get("messages") or []
    post = (ctx.post_values or {}).get("messages") or []
    if len(post) < len(pre):
        return post  # unusual: state replaced
    return list(post[len(pre):])
```

No public API change. No data-model change. Preserves all prior semantics
(cache tokens fold into prompt; reasoning_tokens sub-field of completion;
`None` for non-LLM nodes).

## Finding #2 — Handoff tool calls indistinguishable from domain tool calls (informational)

`create_handoff_tool` produces regular tool_call events (`transfer_to_bob`, `transfer_to_alice`). Chronos records them faithfully. Users running `chronos diff` across handoff nodes will see them as "normal" tool calls with a specific name pattern. Not a bug — this is how langgraph-swarm-py implements handoffs — but worth documenting so users don't hunt for a special event type that doesn't exist.

## Finding #3 — `Node` has `state_after` only, no `state_before` (friction, not bug)

Writing `inspect_trace.py` I kept reaching for `node.state_before` and having to walk back to the prior node's `state_after`. Documented pattern in case study; decided against schema change — storage duplication not worth the 3-line convenience. `chronos diff` already does this internally.

## 2. Deliverables

### 2.1 `src/chronos/adapters/linear/` (new package)

- **`__init__.py`** (~30 LOC): re-exports `LinearRuntime`, `LinearRecorder`, `AdapterError`.
- **`recorder.py`** (~385 LOC):
  - `@dataclass(frozen=True) class LinearRuntime`: immutable `list[tuple[str, Callable[[dict], dict]]]` with optional per-step `NodeKind` map. Rejects duplicate step names at construction (ADR-014 R1 hidden requirement: node identity must be unique within a run — matches `LangGraphRecorder.node_identity_resolver` default). Helper: `step_index_of(name) -> int`.
  - `class LinearRecorder`:
    - `record(runtime, *, thread_id, initial_state=None, task=None, tags=None) -> AbstractContextManager[RunRef]`: pre-generates run_id + parent-chain node_ids inside the CM, executes `runtime.steps` inline, captures `state_before` / `state_after` per step, pops `__chronos_usage__` hint into `Node.usage`, persists `Run + Nodes` in a single `store.transaction()` on normal exit. On exception: persists a zero-node `status=FAILED` Run shell for forensic visibility, then re-raises.
    - `fork(runtime, *, parent_run_id, at_node_id, overrides, child_thread_id, task=None, tags=None) -> AbstractContextManager[ForkRef]`: validates (parent exists, at_node_id belongs to this parent, child_thread_id ≠ parent's, runtime matches the one used for parent — by comparing node name sequence); seeds child state = parent node's `state_after` + `overrides`; re-executes `runtime.steps[fork_index+1:]`; persists `Run + Nodes + Fork` in one transaction on exit. Same failed-shell behaviour on step exception.

## 4. Design Decisions

### 4.1 `LinearRuntime` as a frozen dataclass

Chose `@dataclass(frozen=True)` over a plain `list` or a builder class:

- Frozen → immutable → safe to hash for `inspect.signature` protocol conformance tests.
- Dataclass → `__repr__` / `__eq__` for free → `assert runtime1 == runtime2` works in tests.
- `kind_map: dict[str, NodeKind] | None = None` optional → sane default (`fn` for everything) but extensible.

Alternative considered: a `typing.Protocol class RuntimeProto: def step_names() -> list[str]`. Rejected — ADR-016 intentionally does **not** put a `RuntimeProtocol` into the contract (one of the five rejected alternatives in ADR-016 §Rejected). Matching LangGraph's "runtime is duck-typed" conformance keeps the story consistent.

### 4.2 Usage hint via `__chronos_usage__` state key (not a separate callback)

The LangGraph adapter extracts usage via a callback registered on the LangGraph runtime. A linear adapter has no runtime-level hook, so the contract has to flow through state. Chose: step functions **may** return a dict containing a reserved key `__chronos_usage__: UsageResult`. On capture, `LinearRecorder` pops this key into `Node.usage` and removes it from `state_after` to keep the state-diff view clean.

Benefits:

- ADR-015 Layer 1 invariant upheld (`UsageResult` dataclass; adapter doesn't synthesize its own shape).
- Zero new Protocol required (the mechanism is entirely internal to the linear adapter).
- Duck-typed step functions stay ordinary `dict → dict` lambdas in tests and demos.

Cost:

- Users must know the reserved key. Documented in `__init__.py` module docstring + CHANGELOG entry.
- If a step legitimately uses `__chronos_usage__` as a domain key, there's a collision. Mitigation: the double-underscore prefix is a strong "reserved by framework" convention; acceptable.

### 4.3 Fork by re-execution (not checkpointer)

This is the R28 contribution to validating R27 risks-doc **R-2** (*fork primitive fundamentally non-portable*). The mitigation strategy in R-2 was: *"ADR-016 `fork()` specifies postcondition only — child run starts from parent `state_after` + overrides — not mechanism."*

R28 proves this is workable:

- LangGraph adapter uses `SqliteSaver` checkpointer + `graph.invoke(None, thread_id=...)` to resume.
- Linear adapter re-executes `runtime.steps[fork_index+1:]` with seeded state.
- Both produce the same observable `Fork` row with `child_run_id`, `node_ids`, `state_after` matching the Protocol contract.

R29 dogfood will turn this into a CI test. R29 verdict on R-2 expected: ✅ confirmed.

### 4.4 "Runtime mismatch" check for fork

Given that fork doesn't have direct access to the runtime used for the parent run (the parent run was recorded by *some* recorder with *some* runtime, and we're given only `parent_run_id` + `at_node_id`), the Linear adapter validates conformance by: loading parent's Node rows by name, then asserting `[n.name for n in parent_nodes] == [s[0] for s in runtime.steps]`. If names differ, raise `AdapterError("runtime mismatch")`.

This is a **best-effort structural check**, not a semantic guarantee. A future version could store `runtime_fingerprint` on the Run row (hash of `(name, source_location)` pairs) for stronger guarantees. Out of scope for R28; candidate for ADR-017.

## 5. Lessons

### L1 — Structural Protocol conformance needs a runtime test

ADR-016 says "Python Protocols are structural" — meaning `isinstance(rec, RecorderProtocol)` works without explicit inheritance. But in practice: if the Protocol file doesn't exist yet (which is the case — ADR-016 rollout step 2 hasn't happened), there's nothing to `isinstance` against.

Fix: `TestProtocolConformance` uses `inspect.signature` to compare the two concrete Recorder classes directly. This gives *stronger* guarantees than structural typing (catches parameter-kind drift, e.g. kw-only vs positional), and works **without** the Protocol file existing. When ADR-016 rollout step 2 eventually happens at R30, we can additionally add `cast(RecorderProtocol, LinearRecorder(...))` smoke tests, but the signature-equivalence check stays as the stronger one.

### L2 — `ruff --fix` reformats files, invalidating line numbers between chained `patch` calls

When applying lint fixes interactively: `ruff check --fix` edits files in place, changing line numbers. Strategy: after `--fix`, always re-read the file or grep for the context string before the next `patch` call. Don't chain patches off stale line numbers. (Found this the hard way when patching `test_fork_rejects_at_node_id_wrong_parent` right after a `--fix` run — offsets were off by 2.)

### L3 — Usage hint through state needs explicit "pop" semantics

First draft left `__chronos_usage__` in `state_after` after extraction. Test `test_usage_hint_extracted_and_popped_from_state` caught it. Fix: `state_after = {k: v for k, v in raw.items() if k != "__chronos_usage__"}`. Lesson: when a framework attaches metadata to a user-facing data structure, always decide explicitly whether it's additive (visible in diffs) or transparent (popped before persistence). For usage hints, transparent is correct — usage belongs to `Node.usage`, not to the state trail.

### L4 — ADR-016 rollout step 2 (Protocol file) is pre-existing tech debt

R28 does **not** create `src/chronos/adapters/protocols.py` (ADR-016 rollout step 2 prescription). Reason: that refactor touches `langgraph.py` re-imports too, which would inflate R28 scope. Noted in CONTEXT.md §5 and in R30 candidate section as a cleanup to fold into the v0.2.0-alpha release bundle. This is the right place to do it — release rounds are the natural time for small cross-cutting tidy-ups.

### L5 — `failed shell Run` semantics are consistent across adapters and should go into ADR-016

Both `LangGraphRecorder` and `LinearRecorder` persist a zero-node `status=FAILED` Run on exception before re-raising. ADR-016 currently says only "atomicity" as a lifecycle invariant, without specifying what "atomic" means on failure. Candidate addition in R29 or R30: add "Invariant F1: on step exception, recorder MUST persist a `status=FAILED` Run row (possibly with zero or partial Node rows) before re-raising. This enables forensic analysis of what the user attempted." Filed as a potential R29-prime (ADR-016 errata).

## 2. The forcing decision: sync wrap vs async Protocol

`autogen-agentchat==0.7.5`'s top-level surfaces (`Team.run`, `Team.run_stream`, `AssistantAgent.on_messages`) are **all async** with no sync equivalents. `RecorderProtocol.record()` (ADR-016 P1) is a sync `AbstractContextManager`. A user cannot naively `await team.run(...)` inside `with recorder.record(...) as ref:`.

Per CONTEXT.md §6 cron directive: "if you hit async-first API incompatibility, stop and write ADR-017 before bending the Protocol." I stopped.

### 2.1 The 3-minute spike that changed the answer

Before writing the ADR I ran `/tmp/spike_autogen.py` — a stub `ChatCompletionClient` + stub team + `asyncio.run(team.run(task="..."))`. Result: the coroutine returns a `TaskResult` containing `messages: list[BaseChatMessage]` covering the entire multi-agent exchange. **Async streaming is not needed to see nodes** — post-hoc `result.messages` inspection is enough for Phase 2 scope.

This killed the assumption baked into the CONTEXT.md §6 wording ("async-first API → async Protocol family needed"). The mismatch is API-surface-deep, not architecturally deep.

### 2.2 Why Path A (sync wrap) won

Four reasons, in order of weight for a GitHub breakout OSS project:

1. **DX is the north star**. Users write `asyncio.run(team.run(...))` — one idiom they already know — not flip every call site to `async with`.
2. **Single Protocol family = single audit surface**. Two parallel Protocols double every ADR-015/016 invariant.
3. **Spike proved `TaskResult` is enough**. Streaming only buys progressive observation (a Phase 3+ UI concern, explicitly deferred in ADR-014).
4. **Path B remains available as a strict superset later** if streaming/cancellation needs demand it.

Rollback plan is in ADR-017 §Consequences: if Phase 2 dogfood reveals `asyncio.run()` too painful (FastAPI/Jupyter loop-already-running), add `AsyncRecorderProtocol` in v0.3 as a superset.

## 3. Design notes

### 3.1 `submit_result` hook

Since execution happens in user code (not the adapter), the recorder needs a way to see the `TaskResult`. Two channels, in priority order:

1. **`ref.submit_result(result)`** — explicit, documented primary path. Guaranteed correctness.
2. **`runtime.messages`** — fallback. AutoGen `Team` keeps the history buffer as an instance attr after `run()` completes. Used when the user forgets to call `submit_result`.

If both are absent, we persist a Run with zero nodes (visibility over silent success). This is the `_resolve_messages()` helper in `recorder.py`.

### 3.2 Message → NodeKind mapping (overridable)

| AutoGen class | Default NodeKind | Notes |
|---|---|---|
| `TextMessage` (source=user) | `FN` | Demoted from `LLM` — human input, not model output |
| `TextMessage` (source=assistant) | `LLM` | |
| `ToolCallRequestEvent` / `ToolCallExecutionEvent` / `ToolCallSummaryMessage` | `TOOL` | |
| `HandoffMessage` | `ROUTER` | Agent→agent delegation |
| `MultiModalMessage` | `LLM` | |
| `StopMessage` | `END` | |
| anything else | `FN` | Conservative default |

The `_kind_map` on `AutoGenRecorder` is merge-over-default; users override only the classes they care about.

### 3.3 Usage

AutoGen's `BaseChatMessage.models_usage` is a `RequestUsage(prompt_tokens, completion_tokens)`. The recorder copies these verbatim into `chronos.core.models.Usage`; `reasoning_tokens` stays 0 (AutoGen doesn't currently report it). Simpler than LangGraph's callback-based ADR-015 Layer 4 bridge.

Factory `build_recorder(usage_extractor=...)` raises `AdapterError` to make it loud — the ADR-015 callback path is NOT wired in for AutoGen, and silently ignoring a passed extractor would mislead the caller.

### 3.4 `state_after` for a message-based framework

LangGraph hands us a full state dict per snapshot; AutoGen doesn't. We synthesize `state_after = {"messages": [...cumulative serialized messages...]}`, which matches AutoGen's own model (message history **is** the state). Each Node's `state_after["messages"]` contains every message from step 0 up to and including this step.

### 3.5 Fork not implemented

`AutoGenRecorder.fork()` exists so the class structurally conforms to `RecorderProtocol` but raises `AdapterError("AutoGenRecorder.fork() is not implemented in v0.2.x — tracked in roadmap Phase 3. See ADR-017 §Decision.")`. Testing this path is one of the 10 unit tests; it asserts both the exception type and that "ADR-017" appears in the message.

## 4. The one real bug: `INSERT OR REPLACE` + `ON DELETE CASCADE`

First pass of `_persist_run_and_nodes()` followed the LangGraph pattern: write a `RUNNING` Run, insert Nodes, then re-write the same Run with `COMPLETED` status and `final_state`. Tests failed with `get_nodes_for_run(run.id) == []`.

Root cause: `SqliteStore.put_run()` uses `INSERT OR REPLACE`, which is "DELETE then INSERT" at the SQLite level. `nodes.run_id REFERENCES runs(id) ON DELETE CASCADE` (schema line 49). The second `put_run()` cascade-deleted every Node we'd just inserted.

Fix: build the full final state + cumulative serialized messages **before** the transaction, write the Run exactly once as `COMPLETED` with `ended_at` and `final_state` set up front, then insert Nodes. If any Node insert raises, transaction rolls back and no partial state leaks. Long comment in `recorder.py::_persist_run_and_nodes` documents the pitfall for future adapters.

Lesson for future adapters: **never `put_run()` twice in the same transaction**. If you need to update a Run's status mid-flight, add an `update_run_status()` store method that doesn't cascade.

## 5. Test strategy: duck-typed stubs, no real SDK

Unit tests **do not import `autogen_agentchat`** — they use a `_StubMessage` dataclass factory (`_make_msg("TextMessage", source=..., content=...)` returns an instance whose `type(msg).__name__ == "TextMessage"`) + a `_StubTaskResult` + `_StubTeam`. This matters for two reasons:

1. CI doesn't need autogen as a dependency to run the core test suite (it's an optional dep).
2. The recorder's dispatch logic (kind-mapping, serialization, usage extraction) is tested in isolation from AutoGen version churn — these tests won't break when 0.8 ships.

A separate integration-lane test using a real `Team.run()` is NOT in R33 — it's a candidate for the future AutoGen dogfood round (tracked under Phase 2's `dogfood triple on autogen` in roadmap R3 criterion, which is currently LangGraph + Linear only).

## 6. What R33 did NOT do

- No fork support for AutoGen (ADR-017 §Decision — Phase 3 candidate).
- No AutoGen-side extension of the dual-adapter dogfood (R29's `tests/integration/test_dual_adapter_dogfood.py` stays on LangGraph + Linear; adding AutoGen requires fork, which we don't have).
- No changes to `RecorderProtocol` or `AdapterProtocol` (single Protocol family preserved).
- No changes to ADR-015 extractor contract (AutoGen bypasses it).
- No CLI surface for `chronos autogen <thread_id>` — the existing `chronos` CLI is adapter-agnostic at the store layer and doesn't need per-adapter commands.

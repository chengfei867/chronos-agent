## 3. DX finding ‚Äî `new node ids: []` (the checkpointer gotcha)

The child run is registered and has a `fork_id`, but **zero new nodes**. Root cause is a LangGraph semantic, not a Chronos bug:

- `recorder.fork(...)` seeds the child thread's state via LangGraph's `update_state` at the fork point.
- The stub then calls `graph.invoke(None, {"configurable": {"thread_id": child_thread_id}})`. In LangGraph, `invoke(None, ...)` means "resume from the checkpointed state at this thread." If the graph doesn't have a checkpointer bound, there's nothing to resume from ‚Äî the call returns immediately without stepping any node.
- `dogfood_baseline.build_supervisor_app()` **does** compile with `InMemorySaver`, but it's a **fresh `InMemorySaver` per call**. So every time the stub imports and calls the factory, it gets a brand-new empty checkpointer. The parent run's checkpoint state never reaches it.

**Implication for R22/R23**: the stub today can exercise the Chronos fork-record path and the kwargs wiring, but an end user who just wants the child run to actually *proceed through graph nodes* needs to bind a **persistent checkpointer that the parent run also used** ‚Äî typically a `SqliteSaver` pointing at the same DB, or a shared `InMemorySaver` that's kept alive across `record(...)` and `fork(...)`.

The R22 stub doesn't communicate this. Neither does the `to_python()` docstring or the `fork plan` CLI help.

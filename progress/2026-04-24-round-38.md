## Observations & decisions

### Legend panel (R38-A)

Clean pass — new `components/Legend.tsx` ~200 LOC, `<Panel position="top-right">` collapsible via chip button, i18n strings in `legend.*` namespace. Shipped at 8.5/10 per vision, no controversy.

### Edge selection (R38-B) — the messy part

First draft used AntD `message.info` toast on edge click. **Blocked**: AntD v6 App context `message` static methods return a `useMessage()` instance but the actual DOM notice doesn't render when called from inside a ReactFlow `onEdgeClick` — turned out to be a React context boundary issue with the `<ReactFlowProvider>` subtree. Burned 4 iterations trying to make it work; gave up.

**Pivoted** to a `selectedEdgeId` state + floating `<SelectedEdgePanel>` in `<Panel position="top-left">`. This also exposed and fixed a pre-existing bug: edges in `layout.ts` didn't carry `data.kind` metadata — the selection logic had to infer kind from edge ID prefix string parsing. Added explicit `data: { kind: "sequential" }` / `data: { kind: "fork", childRunId }` on edge construction.

### Visual calibration — the loop

Started with "dramatic" highlight: 4.5px stroke, triple-layer drop-shadow, dash-flow animation, `#c8e2ff`/`#f0c4ff` brighter accent variants. User feedback across 3 rounds landed on:

- **Too bright, too thick** → back to original palette `#58a6ff`/`#c678f7`, 2.6px stroke, single 6px glow at 0.55 opacity.
- **Dash-flow animation "violates"** → dropped the `chr-edge-dash-flow` keyframe entirely. Clashed with fork edges' natural dashed style.
- **Unselected 0.5 opacity reads as "broken"** → lifted to 0.65.
- **SelectedEdgePanel gradient + backdrop-blur + 3-layer shadow too much** → simplified to single drop-shadow + left-border accent.

### Vision arbitration — new working rule

`browser_vision` gave inconsistent scores across cosmetically similar screens: 7.5 → 6.5 → 5.5 for three builds where the changes were tiny increments. Cross-checked the same cached screenshots via `vision_analyze` (different LLM call path) and got 8-8.5 on each. The tool is flaky enough to gaslight iteration.

**New rule**: when `browser_vision` output is ambiguous or conflicts with DOM self-check, stop arguing — ship the screenshot to the user with 3-4 focal questions and let the human arbitrate. Cut R38 polish loop from estimated 7+ iterations to 1.

### UI polish priority — user correction

User feedback closed the loop on a R36-D misread: "组件库级精致度不是 nice-to-have 是 blocker" (R36-D) got revised to *"只要不是 bug、能力上确实、交互上逻辑有问题其实都还好，重点保证功能正常，样式的问题没有那么重要，只要够流畅就好"* (R38). Calibrated to:

- **Correctness > flow > visual** — stop magnifying details once functional + smooth.
- **Muted > loud** — user actively dislikes bright-glow / thick-stroke / dash-flow "精致" elements.

Saved both to user profile.

Built the **LangGraph recorder adapter** that converts a completed graph run
into canonical Chronos `Run` + `Node[]` records with zero manual reshape
from the user.

## What shipped

### 1. Spike 4 — empirical probe of `StateSnapshot` (tests/spikes/spike4_probe_snapshot.py)
- Built a 4-node graph, invoked it, then dumped every field of every
  snapshot returned by `graph.get_state_history()`.
- **Key empirical findings** (captured in ADR-004):
  - 6 snapshots for 4 executed nodes (N+2 shape: input placeholder + per-step + terminal)
  - `metadata["writes"]` is **always `null`** on persisted checkpoints —
    the Round 3 CONTEXT.md note about harvesting from `writes` was wrong
  - `snapshot.tasks[0].name` on a pre-execution snapshot tells you which
    node is *about to run* — this is the key for mapping snapshots to nodes
  - `parent_config.checkpoint_id` forms a clean causal chain
  - `created_at` is a parseable ISO-8601 UTC string
  - Terminal snapshot has `next=()` and `tasks=[]` — the final state, not
    a separate Node

### 2. ADR-004 (docs/decisions/ADR-004-langgraph-snapshot-mapping.md, 10k)
- Full algorithm for mapping `list(get_state_history())` → `Run` + `Node[]`
- Documents what we *cannot* capture: token/usage/cost (LangGraph doesn't
  expose them; deferred to M2)
- `kind_map` injection point as the pluggability seam for typed nodes
- 3 alternatives explicitly considered and rejected with rationale
- Open questions & test plan as §Closing

### 3. `src/chronos/adapters/langgraph.py` (336 lines)
- `LangGraphRecorder(store, kind_map=...)` — reusable, stateless between runs
- `recorder.record(graph, thread_id=...)` — context manager yielding a
  `RunRef` that gets populated on exit
- Correctly handles the failure path: if the user's `graph.invoke()` raises,
  we still persist the partial run with `status=FAILED` before re-raising
- Defensive error on LangGraph version drift: `source='input'` check fires
  an `AdapterError` with an actionable message
- Pure post-run reconstruction — no checkpointer subclassing, no mid-run hooks

### 4. Unit tests (tests/unit/test_adapter_langgraph.py, 11 tests)
Using hand-crafted `FakeSnapshot` / `FakeGraph` duck-types (no real
LangGraph import in unit tests — isolates the contract):
1. Top-level Run fields (initial/final state, status, metadata)
2. One Node per executed step, in order, with correct step_index
3. Linear parent_node_id chain
4. State accumulates correctly across nodes
5. `kind_map` honoured (per-node override + default)
6. Checkpoint IDs preserved in node metadata
7. Failed run → `status=FAILED` (still persisted, exception re-raised)
8. Empty history → no-op (no DB pollution)
9. Missing `source='input'` → `AdapterError` (version-drift guard)
10. Single-node graph edge case
11. Cyclic graph (same node_name at different step_index)

### 5. E2E: rewritten `tests/integration/fixtures_writer.py`
- **40 lines of reshape → deleted.** Writer is now 95 lines (was 137).
- Uses `LangGraphRecorder` + `kind_map={n: NodeKind.LLM for n in NODES}`
- All 3 integration tests still green in cross-process subprocess setup

### 6. E2E test assertions updated
- Removed assertions on `usage` / `model_name` / `cost_usd_cents` that

## Engineering lessons

### Spike-before-implement WORKED.
My Round 3 CONTEXT.md assumption that node data lives in
`metadata["writes"]` was **wrong**. If I had skipped spike4 and gone
straight to adapter implementation, I would have written ~200 lines of
code against a false contract, hit confusing test failures, and spent
2+ hours debugging before realising the shape was different. The
60-line probe script saved a multi-hour teardown. **This confirms the
"one spike per non-trivial new integration" policy is worth the cost.**

### Test-contract isolation via duck types.
Unit testing the adapter against `FakeSnapshot` (a 6-field dataclass)
instead of real LangGraph means:
- Tests are <200ms total (no graph compilation / LLM calls)
- The test file *is* the adapter's contract documentation
- If LangGraph changes its API, the integration test will catch it;
  the unit tests will continue passing and point to exactly which
  duck-typed field differs from reality.

### API surface discipline.
Resisted the temptation to fake `usage` / `model_name` fields in the
adapter just to make old E2E assertions pass. Accurately reporting
what LangGraph gives us (and what it doesn't) > superficial green
tests that lie about capability. Deferred to M2 with a clear seam
(the `kind_map` parameter pairs naturally with a future
`usage_harvester` parameter).

## Round 5 candidates (priority-ordered)

1. **M1.5 — Fork primitive**: `chronos fork <run_id> --at-node <step>`.
   Implements the fork side of ADR-003's `forks` table. Requires
   `graph.update_state(as_node=...)` investigation — likely another spike.
2. **CLI polish**: `chronos list`, `chronos show <run>`, `chronos diff <run_a> <run_b>`.
   Mostly read-side; unblocks user-facing smoke-testing.
3. **Usage harvester hook**: optional `usage_extractor: Callable[[dict], Usage|None]`
   on the recorder. Lets users embed usage in state and have the adapter
   pick it up. Low-effort, closes the biggest "what's missing from v0.1"
   objection.

Leaning toward **M1.5 next** — forks are the headline feature and without
them the persistence layer has no downstream consumer justifying its
existence.

## TL;DR

Shipped the **fork primitive** on `LangGraphRecorder`. Users can now:

```python
with recorder.fork(
    graph,
    parent_run_id=parent.run_id,
    at_node_id=research_node.id,
    overrides={"research": "alternative content"},
    child_thread_id="fork-t",
    reason="try alt research",
) as fref:
    graph.invoke(None, {"configurable": {"thread_id": "fork-t"}})
# fref.child_run_id / fref.fork_id populated; Fork row linking parent→child is persisted
```

This is the headline feature of Chronos — without it, the tool is just a
trace viewer. With it, we have the "git rebase -i for agents" primitive.

## What shipped

### 1. Spike 5 — empirical probe of forked-thread history (`tests/spikes/spike5_probe_fork_history.py`)

Compared thread A (original `graph.invoke(state, cfg)`) vs thread B
(seeded via `graph.update_state(cfg, values, as_node=X)` + `graph.invoke(None, cfg)`).

**Key findings** (Round 4's adapter assumed A's shape — B is different):

| Property | Thread A | Thread B (forked) |
|---|---|---|
| Snapshot count | N + 2 (7 for 5 nodes) | **N + 1** (4 for 3 downstream nodes) |
| `metadata.source` sequence | `[input, loop, loop, ...]` | `[update, loop, loop, ...]` |
| `metadata.step` | `[-1, 0, 1, ...]` | `[0, 1, 2, ...]` (**thread-local, restarts**) |
| First snapshot's `tasks` | `[]` (placeholder) | `[<next-downstream-node>]` (already "pre-run") |
| Index 0 semantics | Input placeholder | **Seed AND pre-first-downstream** (dual role) |

Round 4 adapter asserts `source == 'input'` → rejects forked threads.
Adapter had to be generalised.

### 2. ADR-005 — Fork Semantics (`docs/decisions/ADR-005-fork-semantics.md`, 12k)

- Full algorithm for `fork()` — pre-flight validation, seeding,
  snapshot walk with `loop_start=0` (vs `=1` for original runs),
  cross-Run `parent_node_id` on the first child node (lineage
  pointer across Runs), and the `Fork` record write.
- 3 alternatives considered (deep-copy checkpointer / full re-invoke /
  framework-agnostic ops layer) and rejected with rationale.
- Documents edge cases (fork from terminal, empty overrides, re-fork,
  forking already-forked runs) and the failure-path contract.
- Open questions (determinism / side-effects) deferred to Phase 3 with
  an explicit v0.1 caveat.

### 3. Refactored `src/chronos/adapters/langgraph.py` (+176 lines net)

- **New**: `ForkRef` dataclass — mirror of `RunRef` with
  `child_run_id` / `fork_id` / `node_ids` fields.
- **New**: `LangGraphRecorder.fork(graph, *, parent_run_id, at_node_id,
  overrides, child_thread_id, reason=None, ...)` context manager.
- **Refactored**: Extracted shared core `_build_run_and_nodes(...)`
  from the old `_persist_from_history` so `record()` and `fork()`
  share 80% of the snapshot-walk code. Parameterised via:
  - `loop_start` — 1 for original, 0 for fork
  - `first_step_index` — 0 for original, `parent.step_index + 1` for fork
  - `first_parent_node_id` — None for original, `at_node_id` for fork
    (cross-Run pointer — the first child node's parent IS the fork point
    in the parent Run; this encodes fork causality in the node DAG)
  - `extra_metadata` — fork adds `forked_from_run`, `forked_at_node`, etc.
- Round 4's `record()` behaviour preserved; all R4 tests still green.

### 4. Unit tests — `tests/unit/test_adapter_fork.py` (9 new tests)

Duck-typed `ForkFakeGraph` simulates `update_state` + `invoke(None)`
without importing real LangGraph, same strategy as R4's `FakeGraph`.

1. Fork persists child Run + Nodes + Fork record, all linkage correct
2. Child nodes: correct order, step_index, cross-Run parent pointer
3. Empty overrides allowed (pure replay mode)
4. Unknown parent_run_id → `AdapterError`
5. at_node_id belongs to different Run → `AdapterError`
6. child_thread_id == parent thread_id → `AdapterError` (overwrite guard)
7. Unexpected `source` on child thread → `AdapterError` (drift guard)
8. User invoke() raises → child still persisted with `status=FAILED`,
   Fork row still written, then re-raised
9. Fork from terminal node → child Run with 0 nodes + Fork record
   (coherent no-op, no crash)

### 5. Integration test — `tests/integration/test_fork_e2e.py` + `fork_writer.py`

- Subprocess runs the spike1 5-node LangGraph (plan → research → draft
  → critique → polish) with real `InMemorySaver`.
- Records parent, forks at research with `{"research": "[FORKED] ..."}`,
  lets the 3 downstream nodes re-execute.
- Main process reopens the DB fresh, asserts:
  - Both runs present (`parent_id` + `child_id` in `list_runs()`)
  - Child metadata shows `forked_from_run`, `overrides_keys`, `fork` tag
  - Child nodes are `[draft, critique, polish]` with `step_index=[2, 3, 4]`
  - First child node's `parent_node_id` = parent's research-node id
  - `Fork` row has the right `edited_fields`, `reason`
  - `get_fork_for_child(child_id)` returns the Fork (reverse lookup)
  - `parent.final_state["final"] != child.final_state["final"]`
    (fork actually diverged)

**Note during writing**: first iteration of `fork_writer.py` had each
node's step closure read only `state["task"]`, so changing `research`
downstream had no effect — final outputs matched! Fix: each step now
reads all upstream fields into its user prompt, so draft/critique/polish
are sensitive to upstream `research` changes. This wasn't a bug in the
adapter — it was a bug in the *test fixture's graph*. Good catch that
the assertion `final_a != final_b` caught it loudly.

## Engineering lessons

### Spike-before-implement **paid off again**

R4's wrong assumption about `metadata["writes"]` was caught by spike4
before implementation; this round, spike5 caught the
`source='update'` vs `source='input'` divergence and the N+1 vs N+2
snapshot count *before* I wrote a single line of fork code. If I'd
just coded against "fork looks like record", I'd have hit the
`AdapterError: expected source='input'` on the first E2E, then spent
30+ min tracing it. **Two for two — keep this discipline.**

### E2E caught a fixture bug

The integration test caught that my sample graph was insensitive to
upstream state changes (each step only read `task`). A unit test with
duck types *cannot* catch this — it's a property of the actual DAG
execution semantics. This vindicates the "duck-unit + real-integration
both" policy from R4. Keep both tiers.

### The shared `_build_run_and_nodes` helper is the right shape

Before factoring: I'd have two nearly-identical 50-line methods. After
factoring: one 40-line method parameterised by 4 small knobs
(`loop_start`, `first_step_index`, `first_parent_node_id`,
`extra_metadata`). When AutoGen adapter lands in Phase 2, the same
helper can serve it too if AutoGen exposes a similar checkpoint-list
abstraction.

## Round 6 candidates (priority-ordered)

1. **CLI read-side commands** (M1.5/M1.6 in roadmap):
   - `chronos runs list [--limit N]`
   - `chronos runs show <id>` — tree of nodes with rich rendering
   - `chronos forks show <fork_id>` — parent Run + fork point + overrides
     + divergence summary
   - Uses the store we already have; low risk, high user-visible value.
   - Makes the tool touchable for the first time.

2. **Structural diff (M1.7)**:
   - `chronos diff <run_a> <run_b>` — align nodes by name, show
     state-after deltas. Particularly valuable for fork parent vs child.
   - ADR-006 will be needed for the alignment algorithm.

3. **Fork API convenience wrappers** — top-level `chronos.fork(...)` that
   constructs the recorder for you. Nice-to-have; the recorder-level API
   is already usable.

4. **Usage harvester hook** — still on the backlog; less valuable than
   getting CLI in front of users.

**Leaning CLI next (1)** — the DB and fork primitive are both working,
but nothing visible to humans yet. One round of `rich`-formatted output
changes the dogfood story from "run pytest to see it works" to
"`chronos runs show <id>` prints a tree". That's a qualitative
improvement in how maintainable and demonstrable the project is.

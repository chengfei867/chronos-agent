# Round 22 -- `fork plan --emit python` (ADR-013 deferred alt C shipped)

**Date**: 2026-04-23 evening CST (R21 done ~20:20, R22 picked up at ~21:18)
**Version at entry**: v0.1.4 + `[Unreleased]` R21 section
**Version at exit**: **v0.1.5** (R21 + R22 bundle)

## Why this round

R21 wrote ADR-013 "fork auto-execution stays frozen" and listed a
deferred alt C: "Chronos emits pastable glue code instead of executing
the graph." That alt was the only one on the deferred list with real
user value attached, not just "maybe someday." R22 shipped it.

The pitch: after three rounds of dogfood the pain point that kept
surfacing was NOT "I want Chronos to run my graph" (execute-fork, ADR-008
rejected, ADR-013 frozen) but "I want to go from the plan JSON to
working code without reading docs." Alt C fills exactly that gap without
crossing the boundary.

## What happened

### R22-A design decisions (5 min)

Two shape options considered:

1. **Standalone subcommand**: `chronos fork emit-python fork_plan.json`
   (reads an existing plan file).
2. **Option on existing subcommand**: `chronos fork plan <run_id> --emit
   python` (single-shot from parent_run_id).

Picked option 2. Rationale:

- New users can `fork plan --emit python` in one step; they never have
  to see the JSON intermediate.
- Users who already have a JSON plan can trivially round-trip via
  `ForkPlan.load_plan(...).to_python()` in Python -- no CLI help needed.
- One fewer subcommand to maintain in the CLI surface.
- If R23+ someone asks for the standalone form, it's a 10-line wrapper
  around the API we already built.

### R22-B implementation (~15 min)

Added `ForkPlan.to_python(*, recorder_var="recorder", graph_var="graph")
-> str` in `src/chronos/fork_plan.py` (+85 LOC, docstring-heavy).

Key design points:

- **Self-contained output**: the generated Python does NOT read the JSON
  plan at runtime. Fork kwargs are inlined as Python literals using
  `repr(value)`. This means the user can pipe the output to a notebook
  cell and run without any Chronos-managed file around.
- **`repr`-based inlining is safe**: `ForkPlan.overrides` already has to
  be JSON-serializable to survive `to_json()`. The set of
  JSON-compatible Python values is exactly the set that round-trips
  through `repr`/`eval` (well, minus `eval` — the output is `compile`d,
  not `eval`d — but as literal source it's fine).
- **Two `TODO(user)` markers**: one for Recorder construction, one for
  graph import. The example Recorder wiring is LangGraph-flavored (the
  only adapter that exists today). When an AutoGen adapter ships, we
  revisit the example — not a blocker for R22.
- **Provenance in docstring**: parent_run_id, parent_node_name,
  parent_node_index, child_thread_id, reason, chronos_version,
  generated_at. So the stub is audit-traceable.
- **Trailing newline contract**: exactly one `\n` at end. Safe to pipe.

CLI wiring (`src/chronos/cli/fork.py` + `src/chronos/cli/__init__.py`):

- Added `emit: str` parameter (default `"json"`) to
  `fork_plan_command`.
- Added `--emit` typer option on the top-level command.
- Back-compat: `--json` flag still emits JSON to stdout (unchanged).
- `--emit python` writes to `--out` (defaults to `./fork_stub.py`
  instead of `./fork_plan.json`).
- Unknown `--emit` values exit 1 with a clear error.

## Lessons

- **`repr()` for Python code-gen**: for any value that must round-trip
  through JSON, `repr()` produces valid Python source. Saves writing a
  custom emitter. The `ForkPlan` schema already enforces JSON-compatible
  types, so this is type-safe by construction.
- **Option 2 vs option 1 instinct**: adding a flag to an existing
  subcommand beats adding a new subcommand *when the inputs are the
  same*. Different inputs → different subcommand. `fork plan` already
  takes `parent_run_id`; `--emit` just switches the output shape.
- **R22 time distribution**: 5 min design → 15 min impl → 5 min tests
  → 5 min release mechanics → 5 min docs. Release mechanics are close
  to free because R13/R16/R19 established the 7-step pattern.

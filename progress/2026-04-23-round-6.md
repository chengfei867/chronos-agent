## TL;DR

Chronos is now **touchable**. Anyone with a `chronos.db` from the adapter
can inspect runs and forks from the terminal without writing Python:

```bash
chronos runs list                     # table of all recorded runs
chronos runs show <run_id>            # tree: nodes + fork-of lineage
chronos forks show <fork_id>          # parent/child + edited_fields
chronos runs list --json              # machine-readable for jq/scripts
chronos runs list --db /path/to.db    # explicit DB
CHRONOS_DB=/path/to.db chronos runs list   # env-var form
```

Every command accepts `--db PATH` / `$CHRONOS_DB` / default `./chronos.db`,
plus `--json` for scripted consumers. Missing DB or missing id exits with
a clear error and non-zero status.

Behind the scenes: the `record()` and `fork()` work from R4/R5 is
unchanged. R6 is purely a read-side facade. Before, verifying the DB
required ad-hoc `sqlite3` one-liners or running pytest; now there's a
first-class user surface.

## What shipped

### 1. Spike 6 — CLI rendering spike (`tests/spikes/spike6_cli_readside.py`)

Before writing a single CLI function, I seeded a SqliteStore with a
parent/child/fork triple and prototyped three rendering paths:

- `rich.table.Table` for `runs list`
- `rich.tree.Tree` for `runs show` and `forks show`
- plain `json.dumps(...)` for `--json`

Caught two issues I would have wasted time on inside the real code:

1. `NodeKind` has no `STEP` variant (thought it did) — the valid kinds
   are `{llm, tool, fn, router, fork, end}`. Adapter steps are persisted
   as `fn`. Fixed in the spike before CLI code ever used it.
2. Mixing rich's Console with `--json` mode is a real hazard — rich
   wraps output at the terminal width. Solution: JSON mode uses plain
   `print(json.dumps(..., indent=2, default=str))`, not `console.print`.

Time spent on the spike: ~10 min. Time it saved: probably 30+ min of
"why does this render wrong in CI but fine locally" debugging.
**Spike discipline: 3 for 3.**

Added:


`typer.Argument(...)` in default arguments. Ruff's B008 is a generally

## Engineering notes

### ADR-006 was **not** needed

The CLI framework / formatting choices are:
- Already fixed by R1 scaffolding (Typer + Rich listed as deps in
  `pyproject.toml` since M1.2, before any ADR).
- Local to the CLI module — no downstream module imports from `cli`;
  changing rendering or even swapping Typer for Click would be a
  contained refactor.
- Not generating any new invariants (schema unchanged, store API
  unchanged, adapter API unchanged).

ADR-006 is reserved for the first real architectural bifurcation that
still matters — the natural next candidate is "diff alignment
algorithm" for M1.7/M1.8, where the choice between step-index alignment
vs node-name alignment vs Myers-style sequence alignment will affect
every downstream diff consumer.

### Observable payoff of R5's cross-Run `parent_node_id`

Until today the "fork point as the first child node's parent_node_id
across Run boundaries" was a storage detail verified only by tests.
Now `chronos runs show <child>` renders `forked from <parent> @ node
<fork_point>` using exactly that pointer (via `get_fork_for_child`).
The first visible dividend on the R5 investment.

### `_open_store` refuses to auto-create missing DBs

Deliberate: `SqliteStore.open()` happily creates a new DB on any path.
For a *read* command, that would silently hide "you're in the wrong
directory" bugs (user sees empty list, not "file not found"). The CLI
wraps `open()` with an existence check and a distinct exit code (2).
Writers (future `chronos record`) will bypass this check.

### `--json` output policy

JSON mode avoids `console.print` entirely and uses `print(json.dumps(...,
default=str))`. This keeps output:
- Greppable / jq-able (no Rich wrapping artefacts).
- Deterministic across terminal widths — important for snapshot tests
  later.
- Safe against future ANSI colour escapes leaking into captured
  pipelines.

`default=str` handles any stray datetime that slipped past the explicit
`.isoformat()` calls — belt-and-braces.

### Tests capture stdout **and** stderr when asserting error text

Typer 0.12 / Click 8 can route `rich.Console.print(..., style='red')`
errors to either stream depending on `CliRunner` config and rich's
tty-detection. Defensive pattern used throughout:
```python
combined = (result.stdout + (result.stderr or "")).lower()
assert "not found" in combined
```
Avoids a flaky "works on my machine" failure mode.

## Round 7 candidates (priority-ordered)

1. **M1.7 — Replay TUI (`chronos replay <run_id>`)** — interactive
   step-through using `rich.live` or `textual`. Pros: the other
   headline feature (record/replay/fork/diff's second letter).
   Cons: TUI eats a lot of round-time and the visual payoff per hour
   is worse than a diff. ADR-007 probably needed (rich.live vs textual
   vs plain curses).

2. **M1.8 — Structural diff (`chronos diff <a> <b>`)** —
   align-by-node-name, show state-after deltas, highlight the
   override-affected subtree. Special case: parent-vs-child-of-fork is
   the killer demo (`chronos diff <parent> <child_of_fork>`). **ADR-006
   needed here (alignment algorithm)**. Small-to-medium scope if we
   commit to "node_name + step_index tiebreak" alignment.

3. **`chronos fork <run> --at <node> --set key=value`** — CLI wrapper
   for the recorder-level `fork()` primitive. Low complexity since the
   primitive already works; glue + KV parsing only. Completes Alex's
   story walkthrough on the CLI level.

4. **`chronos record --graph ... --input ...`** — harder because the
   user's LangGraph lives in *their* Python file, not ours. Probably
   ships as a Python-library API (`recorder = chronos.Recorder(db);
   recorder.record(graph, input)`) wrapping the existing
   `LangGraphRecorder`, not a shell command. See Phase 2 API design.

**Leaning Round 7 = M1.8 diff (option 2)**. Reasoning:
- We *already have* 3 visualisation contexts (runs list, runs show,
  forks show) ready to link a diff view into.
- The fork feature is shipped but has no "see the difference" UI —
  users currently have to eyeball two `runs show` outputs and compare.
  A diff command is the single missing piece that makes the fork flow
  story-complete.
- Replay is cool but Chronos's unique selling point vs Langfuse et al.
  is fork+diff, not replay (competitors already do replay). Diff
  widens the moat; replay narrows the gap.
- ADR-006 is overdue; diff alignment is the best forcing function.

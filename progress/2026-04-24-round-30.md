### 2.3 Latent R29 mypy bug — fixed in R30

**Finding**: R29 CHANGELOG claimed "all existing tests unchanged" and ran `pytest`, but **did not run mypy**. When R30 step-5 full-verify ran `mypy src/`, two errors surfaced in `src/chronos/adapters/linear/recorder.py`:

```
error: Unexpected keyword argument "cost_usd_cents" for "Usage"
error: Unexpected keyword argument "model_name" for "Usage"
```

Root cause: `core.models.Usage` only carries three token fields (`prompt_tokens`, `completion_tokens`, `reasoning_tokens`). `cost_usd_cents` and `model_name` live on **`Node`**, not `Usage`. The R29 generalization of the `__chronos_usage__` hint API had the duck-typed branch passing `model_name=...` and `cost_usd_cents=...` as `Usage(...)` kwargs, which `pytest` silently accepted because pydantic defaults swallow the error at runtime under some configurations.

Fix: the duck-typed and dict branches now extract `model_name` / `cost_usd_cents` into local variables `hint_model_name` / `hint_cost_usd_cents` and pass them directly to the `Node(...)` constructor. The `Usage(...)` call only receives its three legitimate token fields.

## 4. Lessons (L1-L3)

### L1 — Release-cut step 5 (full verify) is cheap insurance

R29 shipped a `pytest`-green change that was mypy-red. The difference between "all R29 commits" and "actually landed release" is one `mypy src/` invocation. The `chronos-release-pattern` skill already lists mypy in step 5; this round is the empirical proof that the order matters (verify BEFORE tag, not after). Added as an invariant in CONTEXT.md's "旧事实" block.

### L2 — `Usage` has three fields, only three, forever

Future adapter authors will reach for `Usage(..., model_name=..., cost_usd_cents=...)` because (a) the LangGraph-adapter `UsageResult` dataclass has those fields right there in the class, and (b) intuition says "usage cost is part of usage metering". Both wrong: the project decided years ago (ADR-013 / R20 Finding #2) that LLM-level metadata lives on `Node`, not `Usage`. The `Usage` docstring even calls this out — R29 ignored it, R30 fixed it. Added as an invariant in CONTEXT.md.

### L3 — A release cut is also an audit round

Bundling six rounds of `[Unreleased]` into one release forced a re-read of every `[Unreleased]` entry, which is how the R29 mypy blind-spot got caught at all. This argues for **occasional multi-round release bundles** rather than one-release-per-round: the aggregation step is a natural checkpoint for cross-round consistency. (Counter-evidence would be: if six rounds accumulate, the audit is slow. R30 took ~20 tool calls, which is fine.)

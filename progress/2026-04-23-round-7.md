## TL;DR

`chronos diff <run_a> <run_b>` is live. For the first time Chronos can
show *why* a fork matters, not just *that* it happened:

```bash
chronos diff <parent> <child_of_fork>              # fork-aware: slices A to post-fork nodes
chronos diff <parent> <child_of_fork> --full       # full-run comparison
chronos diff <a> <b> --json                        # machine output (frozen ADR-006 schema)
chronos diff <a> <b> --verbose                     # inline per-key state_after deltas
```

Alignment is pure `difflib.SequenceMatcher` over the `node_name`
sequence. Node `id` and `step_index` are *display-only* in diff
(ADR-006 §Decision). Fork-aware mode auto-detects the parent→child
fork link and slices `run_a`'s nodes to those at step_index strictly
greater than the fork-point node, because the upstream is identical
by construction and displaying it is noise.

Behind the scenes: ADR-006 locks in the alignment algorithm + JSON
schema so every future consumer (Web UI diff viewer, `chronos
export`, IDE extensions) targets the same contract from v0.1.

## What shipped

### 1. Spike 7 — diff alignment validation (`tests/spikes/spike7_diff_alignment.py`)

Ran 5 canonical cases against `difflib.SequenceMatcher`:

1. Parent vs fork child (common tail) — `ingest REMOVED; rest CHANGED`
2. Early-exit fork (overrides cut run short) — 3 REMOVED + 2 CHANGED
3. Cycles with repeated `node_name`s (router→worker loops) — extra
   iteration correctly tagged as 2 ADDED, others paired by order of
   occurrence
4. Identical runs — all EQUAL (sanity)
5. Structural prefix mismatch — prefix REMOVED, shared suffix CHANGED

All five pass. Decision: stdlib `difflib` is enough; no need for a
custom Myers/affine aligner. This was the crucial empirical check
before committing the ADR — if SequenceMatcher had misaligned the
loop case (case 3) we would have needed a bespoke aligner.

**Spike discipline: 4 for 4** (R4 snapshot mapping, R5 fork history,
R6 CLI rendering, R7 alignment).

### 2. ADR-006 — Diff Alignment Algorithm (`docs/decisions/ADR-006-diff-alignment.md`)

Freezes:
- Alignment key: `node_name` sequence (reinforces ADR-003).
- Algorithm: `difflib.SequenceMatcher(autojunk=False)`.
- Opcode mapping: `equal` → `equal`/`changed` (split on `state_after`
  deep-equality); `replace` → linearised as REMOVE+ADD pairs;
  `delete`/`insert` → REMOVED/ADDED.
- Fork-aware slicing: when B is a child of a fork of A, restrict A to
  nodes with `step_index > fork_point.step_index`. Gated on the Fork
  record being in the DB; unrelated fork records of B are ignored.
- JSON schema frozen — 6 top-level keys, per-entry `tag/node_name/a/
  b/state_diff`.

Rejected: step_index alignment (ADR-005 proves it fails on fork
children), custom Myers (overkill), hash-based alignment (defeats the
purpose), rename-detection on `replace` blocks (Phase 3 concern).

## Engineering notes

### Why `difflib` was the right call (spike 7 receipts)

Going in, the biggest risk was case 3: loops. If SequenceMatcher
treated `[router, worker, router, worker]` as a set rather than a
sequence, repeated-name pairing would collapse and the whole approach
would need Myers. Empirically it works — SequenceMatcher naturally
pairs repeated elements by order because it's a sequence aligner, not
a set operation. This is the single riskiest decision in R7 and it
landed clean.

### `replace` opcode → REMOVE+ADD linearisation

SequenceMatcher's `replace` opcode means "these two blocks don't
align at all — treat block A[i1:i2] as going away and block B[j1:j2]
as new." We deliberately **do not** try to re-pair block members
heuristically ("same kind? same tool_name?"). Every heuristic we
considered is one adversarial fork edit away from being wrong and the
cognitive load on the user reading the output outweighs the benefit.
ADR-006 §Decision 3 pins this; a future Web UI can layer a smarter
pairing on top of the raw output if it wants.

### Fork-aware slicing: the killer demo

For `chronos diff <parent> <child_of_fork>`, naive full-run diff would
produce `{REMOVED: N-k+1, CHANGED: k-1}` where `N` is parent length
and `k` is the fork point — the REMOVEDs are *always there* and make
the real changes hard to see. Fork-aware slicing (default) drops them.
Users who want the full picture pass `--full`.

This is the first time R5's cross-Run `parent_node_id` infrastructure
pays a *differentiated* dividend. In R6 it powered a lineage
annotation line; in R7 it powers an actual semantic distinction
between "here's what's downstream of your edit" and "here's the full
runs side by side."

### `Protocol` instead of real store in diff.py

`_DiffStore` Protocol (3 methods: `get_run`, `get_nodes_for_run`,
`get_fork_for_child`) keeps `core/diff.py` independent of
`store.sqlite`. Unit tests use a `_FakeStore` dict-based
implementation (~20 LOC). This matches the R4/R5/R6 pattern of
duck-typed fake + real store integration side by side.

### `state_after` deep-equality is load-bearing for "equal vs changed"

Two nodes with the same `node_name` and the same aligned position can
still have different `state_after` (the whole point of fork). The
split within the `equal` opcode handler is therefore:

```python
if na.state_after == nb.state_after:
    tag = "equal"
else:
    tag = "changed"
    state_diff = _state_diff(na.state_after, nb.state_after)
```

Pydantic-backed dicts compare value-by-value, so this is O(n) in the
total size of state_after. For real LangGraph states with LLM context
windows, this will eventually need a smarter comparator (content
hashes + lazy drill-down) — noted as Phase 3 work.

### `--verbose` uses `→` (U+2192) not `->`

Deliberate: the rich output is always presented to a terminal with
UTF-8; the CLI tests accept either `→` or `->` for portability of
assertions but the user-facing output is consistent with the rest of
Chronos's Unicode-forward style.

## Round 8 candidates (priority-ordered)

1. **M1.7 — Replay TUI (`chronos replay <run_id>`)** — the last letter
   of record/replay/fork/diff we haven't shipped. Users can now
   `list`/`show`/`diff` but not step through. ADR-007 likely needed

   CLI process doesn't have. Likely resolution: `--graph module:attr`
   dynamic import convention, with an ADR-008 discussing the
   CLI-vs-library split for operations that need user code. Alex's
   story ends here; it's the final piece of the story walkthrough.

3. **M1.9 — Getting Started + `examples/` + v0.1.0 tag** — we're now
   arguably feature-complete for Phase 1 MVP; a polish/docs round is
   warranted. `docs/getting-started.md`, 1–2 runnable example agents
   under `examples/`, real install instructions in README, and cut
   v0.1.0. Low risk, high marketing ROI.

4. **Dogfood: have Hermes itself use `chronos diff` to compare two of
   its own runs from earlier rounds.** The roadmap mandates
   dogfooding before phase exit (§Cross-Phase Commitments §3). Could
   pair with option 3 above.

**Leaning Round 8 = option 3 (docs + v0.1.0 tag + dogfood).** Reasoning:
- Phase 1 MVP is functionally complete: record (M1.4), fork (M1.5),
  CLI read (M1.6), CLI diff (M1.8). The only literal M1.x not shipped
  is M1.7 replay, but as R6 notes observed, replay is catch-up to
  competitors while fork+diff is the moat. Chronos can ship v0.1.0
  without replay and delay M1.7 to a 0.1.1.
- Dogfooding is an overdue roadmap commitment and also the single
  most effective way to catch real-world UX issues in the diff
  command we just shipped.
- Cutting v0.1.0 forces us to actually write the `getting-started`
  doc — a forcing function we've been avoiding.

Alternative: if we want the record/replay/fork/diff story complete
before v0.1.0, Round 8 = option 1 (replay TUI) then Round 9 = option
3 (docs + tag). Both paths are defensible; I'll leave the final call

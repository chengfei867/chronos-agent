## 学到 / pitfall

- **CONTEXT §6 TODO 误记风险 (本轮新发现)**: 上轮 agent 写"下一轮做 X"时如果没先 verify X 是否已存在, 下一个 agent 会按字面意思上手. 规则: **每轮开干前先 `chronos --help` + 对相关 endpoint/命令 grep 一遍, 确认 TODO 不是幽灵**. 成本 60 秒, 收益是一整轮不白干. ADR-018 meta-note 里写了.
- **命名不对称可以是健康的**: CLI 用 `diff` (继承 `git diff` / `diff(1)` 的 30 年传统), API/SPA 用 `compare` (对非开发者更友好), 叙事用 `compare` (读着顺). 强行统一反而更难用.
- **v0.2.0 release 已经 auto-建了**: 我并不知道这是哪轮干的 — releases API 返回 `created_at: 2026-04-24T13:59:24Z`, 对照 git log 是 R38 release cut 同一天, 说明 R38 或 R35-A 收尾时走过 `chronos-release-pattern` skill 的 "POST /releases" 步骤. **下轮检查 release 状态的正确方式**: 先 `curl api.github.com/.../releases` (走直连, 别走 gh-proxy), 别凭 CONTEXT 里的老记录判断.

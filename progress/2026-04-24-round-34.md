# Round 34-A: Local HTTP API (FastAPI read-only server)

**Date**: 2026-04-24 (Beijing time, noon)
**Branch**: `main`
**Status**: ✅ shipped — 363/363 tests green, ruff + mypy clean

## Mission

R34 opens the **Web UI** front of Phase 2. R34-A is the backend-only slice:
a read-only FastAPI app over `SqliteStore` that the eventual `chronos web`
command (R34-B) and any community-built viewer can mount as an HTTP layer.

Four motivating use cases shaped the endpoint list:

1. `chronos web` starting uvicorn + opening a browser to a local URL.
2. A community-built ReactFlow viewer consuming `/runs/{id}/tree`.
3. External dashboards (someone's Grafana, someone's internal tool)
   polling `/runs` to surface "what did the agent do today?".
4. `curl`-based debugging during Phase 3 dogfood.

Out of scope for R34-A (deferred to later sub-rounds):

- The `chronos web` CLI command (R34-B).
- The actual ReactFlow frontend (R34-C).
- Auth, CORS, pagination beyond `limit`, streaming, websockets.
- Any write endpoints — mutations go through adapters, always.

## What shipped

### `src/chronos/api/server.py` (new, ~230 LOC with module docstring)

`build_app(store: SqliteStore) -> FastAPI` — factory returns a fresh app
closed over the given store. **Six endpoints**:

| Endpoint                      | Purpose                                          | 404 shape                   |
| ----------------------------- | ------------------------------------------------ | --------------------------- |
| `GET /healthz`                | Liveness probe + `schema_version` echo           | —                           |
| `GET /runs?limit=N`           | Most recent runs (default 100, 1 ≤ limit ≤ 1000) | —                           |
| `GET /runs/{id}`              | Run + ordered nodes                              | 404 if run missing          |
| `GET /runs/{id}/nodes`        | Nodes only (for paginating UIs)                  | 404 if run missing          |
| `GET /runs/{id}/forks`        | Forks where this run is the parent               | 404 only if run missing     |
| `GET /runs/{id}/tree`         | Neutral reasoning-tree JSON (contract endpoint)  | 404 if run missing          |

**404-strict on the run path** — every `/runs/{id}/...` distinguishes
"no such run" (404) from "run exists but has no nodes/forks" (200 with
`count: 0`). A viewer that treats both as empty would misrender.

### `src/chronos/api/__init__.py` (new, 12 LOC)

Re-exports `build_app` for `from chronos.api import build_app`.

### `src/chronos/store/sqlite.py` (2 changes)

1. **New method** `get_forks_for_parent(parent_run_id) -> list[Fork]` —
   mirrors `get_fork_for_child` on the other side of the fork relation,
   ordered by `created_at ASC`. Needed by `/tree` and `/forks` endpoints;
   cleaner than ad-hoc SQL in the server layer.

2. **`SqliteStore.open()` flips `check_same_thread=False`** on the sqlite3
   connection. FastAPI runs sync endpoints in a worker thread-pool, so
   `uvicorn` (and `TestClient`) dispatch store reads off the thread that
   opened the DB. SQLite's engine is thread-safe in its default serialized
   mode; `check_same_thread` is the **Python-layer** guard, not the
   engine's. We hold a single shared connection in autocommit + explicit
   `transaction()` CM, so flipping this is safe. Inline 10-line comment at
   the `sqlite3.connect()` call documents the rationale for auditors.

## Key design decisions

### 1. Neutral reasoning-tree shape (not ReactFlow-specific)

`/runs/{id}/tree` returns:

```json
{
  "run_id": "...",
  "nodes": [ <full Node dict via pydantic model_dump(json)> ],
  "edges": [
    {"from": "<node_id>", "to": "<node_id>", "kind": "sequential"},
    {"from": "<parent_node_id>", "to": "<child_first_node_id>",
     "kind": "fork", "fork_id": "...", "child_run_id": "...",
     "edited_fields": {...}}
  ],
  "child_runs": [ <full Fork dict>, ... ]
}
```

Rationale: the API contract is framework-neutral. ReactFlow wants
`{id, position, data, type}` on nodes and `{id, source, target}` on
edges — all of which the frontend can derive from our shape, but none
of which we bake in. A future Mermaid / D3 / Cytoscape viewer gets the
same JSON and transforms it to its own shape. This is the single most
load-bearing design choice in R34-A; getting it wrong locks us into
one viewer.

A fork edge whose child run has no nodes yet (e.g. still RUNNING) has
`to: null` — the frontend renders an "unresolved branch" rather than
mis-pointing. `child_runs` is a parallel summary so a UI can lazy-load
child subtrees without re-fetching the parent's full tree.

### 2. Sync handlers + `check_same_thread=False`

First cut had sync handlers but kept `check_same_thread=True` (sqlite3
default). `TestClient` failed immediately with
`sqlite3.ProgrammingError: SQLite objects created in a thread can only
be used in that same thread`, because FastAPI dispatches sync endpoints
via Starlette's `run_in_threadpool` → `anyio.to_thread.run_sync`.

Two resolution paths were considered:

- **Path A (rejected)**: make handlers `async def` to keep store calls
  on the event-loop thread. Works, but blocks the event loop on every
  sqlite read — bad hygiene even for a local single-user server.
- **Path B (chosen)**: flip `check_same_thread=False`. SQLite itself is
  serialized-thread-safe; the Python guard is belt-and-suspenders.
  This is how every popular FastAPI-on-SQLite project on PyPI configures
  its connection. No downside for our usage pattern (one shared
  connection, autocommit, explicit `transaction()` CM).

Path B is cleaner and doesn't block the loop. Path A's thread-pinning
was only ever a workaround.

### 3. `build_app(store)` factory, not module-level `app`

Each call returns a fresh `FastAPI` instance closed over the given
store. Prevents the classic "`app = FastAPI()` at module level +
global state" trap that makes production bindings a pain to test.
`test_build_app_binds_distinct_stores` exercises this — two apps
bound to two stores don't cross-talk.

### 4. Pydantic `model_dump(mode="json")` for all responses

No hand-rolled serialization. Pydantic handles `datetime` → ISO-8601,
`StrEnum` → string, nested `Usage` → dict. If we ever add a field to
`Run` / `Node` / `Fork`, every endpoint inherits it for free.

## Pitfalls hit (lessons for future rounds)

1. **Prior context window's CHANGELOG diverged from actual code.** When
   R34-A resumed after context compression, the CHANGELOG already had
   R34-A entries describing 4 endpoints, ReactFlow-shaped tree, `async
   def` handlers, and `ASGITransport` — none of which matched what I
   actually implemented (6 endpoints, neutral shape, sync handlers,
   TestClient + `check_same_thread=False`). Had to rewrite all three
   changelog sections to match reality. **Rule of thumb post-compaction:
   always `git diff` before committing and cross-check against what
   the code actually does — the summary may describe a plan, not the
   built thing.**

2. **SQLite thread guard was invisible until the first `TestClient`
   call.** Pure-unit tests on store or on `build_app()` routes listing
   never tripped it. Only the first endpoint that actually read the
   store failed. Lesson: always include at least one end-to-end
   endpoint-hits-store test before declaring the harness working.

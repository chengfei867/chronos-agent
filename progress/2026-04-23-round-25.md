## 1. Goal

Resolve **ADR-014 criterion R2** (Phase 2 entry blocker #2): the
`UsageExtractor` contract is defined by evolution across four ADRs
(009 / 010 / 011 / 012) plus the three convenience-extractor
implementations. A future AutoGen adapter author (R26+) would have to
reverse-engineer the contract from the LangGraph implementation, which
is exactly the drift ADR-014 warned against: *"the second
implementation becomes the de-facto spec and drift compounds."*

R25 consolidates the four ADRs into a single **framework-agnostic
five-layer contract** and adds "Consolidated into: ADR-015" breadcrumbs
to the predecessors so future readers land on the authoritative source.

---

## 2. Why now (order argument)

ADR-014's non-binding forecast was R25 = R2, R26 = R1, R27 = R3, R28 = R4.
R2 goes first because R1 (adapter interface ADR) **cites** the
extractor contract as part of the adapter surface. Writing R1 first
would bake drift into the interface.

Doing R2 first also means R26's R1 ADR can reference ADR-015
Layers 1-3 as invariant foundations instead of rederiving them.

Five-layer structure:

| Layer | Scope | What it pins |
|-------|-------|--------------|
| **1 тАФ Data shape** | framework-agnostic, invariant | `UsageContext` / `UsageResult` frozen dataclasses; `pre/post_values` are JSON-safe dicts (Layer 3 guarantee); `pre/post_snapshot` are adapter-native `Any` |
| **2 тАФ Protocol & lifecycle** | framework-agnostic, invariant | 6 lifecycle rules: optional kwarg (default None = no-op, backwards compatible); one call per node; called AFTER coercion / BEFORE append; `None` return = non-LLM node (expected); **raised exception = warning + skip, NEVER abort recording**; field mapping verbatim (recorder does not synthesize values) |
| **3 тАФ Serialization boundary** | framework-agnostic, invariant | `_jsonable` recursive algorithm (primitives / `model_dump()` / `__dict__` / containers / `repr()` fallback тАФ never raises); adapterтЖТstore invariant, not LangGraph-specific |
| **4 тАФ Multi-call-per-node delta** | policy invariant, SHAPE framework-specific | Must sum usage across all new AIMessages between pre- and post-snapshot (not "last wins"). ## 5. What surprised me / what I learned

**F1 тАФ The "who cites ADR-015" part was the easiest.**
Once the five-layer structure was in place, writing the portability
matrix (which layers AutoGen honors verbatim vs. specializes) almost
wrote itself, because ADR-011's `_jsonable` algorithm and ADR-012's
delta rule are literally framework-independent тАФ I had been treating
them as LangGraph-specific in my mental model, but they aren't. That's
the drift ADR-014 warned about catching: the existing code had
implicitly correct generalization, but the docs were written as if
LangGraph were the only adapter that would ever exist.

**F2 тАФ Cross-link breadcrumbs > rewriting predecessors.**
Initial instinct was to replace the old ADRs with "see ADR-015."
Rejected: the R15 Anthropic cache-token rationale, the R18 swarm
dogfood evidence with Bob-node numbers, the R12 decision-context for
the original hook тАФ these are *decision history*, not contract spec.
Losing them would erase the "why" and leave only the "what." The
breadcrumb pattern preserves both reachability and history, and is
cheap (one line per ADR). Added this as a durable lesson to CONTEXT.md
┬з5 ("ADR consolidation pattern (R25)").

**F3 тАФ `Consolidated into` is stronger than `Superseded by`.**
The standard ADR vocabulary offers `Supersedes` / `Superseded by`, but
neither fits here. The predecessors are NOT superseded тАФ their
decisions still hold, they just no longer document the contract
*surface*. "Consolidated into" captures this: you read ADR-015 for the
contract, you read ADR-009/010/011/012 for the historical reasons.
Minor vocabulary innovation, likely reusable.

**F4 тАФ The `CONTEXT.md ┬з6 rewrite` patch almost silently dropped a
section header.** The `patch` tool replaced my `### цЧз R23-A шобхИТ`
section header when I swapped the R24/R25 plan blocks, because the
R23-A header and its bullets were adjacent to the replace boundary.
Caught on `grep` verification, fixed with a second small patch. Future
CONTEXT.md rewrites should `grep -n "^### "` before + after to verify
section-header count unchanged unless intentional.

## 7. Next-round hand-off (R26 = R1)

**Target:** ADR-016 тАФ Adapter interface

**Inputs already lined up:**
- ADR-015 Layer 1-3 = half the adapter interface (invariant data shape
  + lifecycle + serialization boundary). Cite, don't rederive.
- `src/chronos/adapters/langgraph.py` (621 lines) = the de-facto
  adapter surface that needs formalizing. Key candidates for the
  protocol: `LangGraphRecorder.__init__(graph, store, usage_extractor=None)`,
  `recorder.record(...)` context manager, `recorder.fork(...)` context
  manager, `recorder._extract_usage(ctx)` lifecycle hook, state
  coercion contract.
- ADR-004 (old LangGraph mapping ADR) = partial prior art; may need
  amending or citing.

**Open design questions for R26 to settle:**
1. Is `AdapterProtocol` a `typing.Protocol` (structural) or an ABC
   (nominal)? Duck typing is the project's preference тЖТ likely Protocol.
2. Does the ADR require a non-trivial adapter change (e.g., subgraph
   nesting support) as proof of interface soundness, or is prose
   sufficient? ADR-014 R1 says "with a non-trivial change
   implementable without touching `chronos.core.*`." Minimal-effort
   interpretation: prose argument + a diff-sketch in the ADR. Maximal
   interpretation: actually implement subgraph nesting in R26 or
   R26-prime. Recommend the prose+diff-sketch version for R26, with
   the actual refactor deferred to R26-prime if needed.
3. How does `ForkProtocol` interact with `RecorderProtocol`? Fork is
   currently `recorder.fork(...)` тАФ is that part of the Recorder
   interface or a separate protocol? Probably separate, because not
   every future adapter will support fork semantics identically
   (AutoGen group chats may not have a clean equivalent).

**Red lines for R26:**
- Don't break v0.1.6 CLI surface.
- Don't `uv add autogen-agentchat` (Phase 2 still closed).
- Don't mutate `chronos.core.*` тАФ that's exactly what R1's
  non-trivial-change criterion is testing for.

## 3. Lessons (L1-L4)

### L1 â€” `parent_node_id`, not `parent_id`
Field name for the within-run node chain is `Node.parent_node_id`. Also exists on `Fork` as `parent_node_id` (the fork point node id). The generic token `parent_id` appears nowhere in `core/models.py`. **Always grep `class Node` before asserting.**

### L2 â€” `UsageResult` is a **frozen dataclass** with only 5 fields
`prompt_tokens`, `completion_tokens`, `reasoning_tokens`, `cost_usd_cents`, `model_name`. **No `raw: dict` field** (I hallucinated one from general OpenAI-shape intuition). Check `langgraph_usage.py` line 51-66 before constructing.

### L3 â€” Zero-dep adapters pay coupling cost via duck typing, not imports
The Linear adapter's docstring says "zero external dependencies." Accepting a `UsageResult` hint without importing `langgraph_usage` requires `isinstance` checks against core types only (`Usage`, `dict`) plus a `getattr`-based duck-typed fallback. This pattern will recur for every future adapter that wants to interop with the LangGraph usage contract without taking the LangGraph dep.

### L4 â€” Dogfood tests belong in `tests/integration/`, not `tests/` root
The project convention (from `tests/integration/test_fork_e2e.py`) puts multi-adapter / slow tests under `tests/integration/`. Placing the R29 dogfood there keeps the unit-test feedback loop fast and matches the implicit pytest discovery hierarchy. No pytest marker needed â€” existing `pyproject.toml` config runs all tests by default.

### L5 â€” "Symmetric assertion helpers" amortize test bloat
`_assert_parent_run_shape(store, run_id, adapter_label)` runs once per adapter inside each scenario and produces adapter-labelled error messages. Without it, Scenario A would have been ~150 LOC of duplicated assertions. Pattern to reuse in R30+ when a 3rd adapter (AutoGen) enters the dogfood.

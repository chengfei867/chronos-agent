## Next steps (R17 candidates)

The landscape after v0.1.3:

1. **Fork execution engine (M2.1)** — `chronos fork run` actually runs
   a forked plan. Needs ADR-011 covering automated-execution safety
   (sandbox, timeout, budget). This is the Phase 1.5 → Phase 2 bridge.
   **Requires user sign-off** (changes ADR-008 frozen boundary).
2. **LangSmith tracer callback extractor** — third LangChain
   usage-accounting path. Smaller scope than R15; no-drama additive.
3. **AutoGen adapter (Phase 2 start)** — **HARD RED LINE**: requires
   explicit user sign-off. R10 was pulled back for this reason; do not
   start without.
4. **Web UI skeleton** — too big for one round, don't cron-start.
5. **Tech debt**: `replay.py` 391L / `fork.py` 367L could be further
   decomposed but not urgent.

**Recommendation for R17**: Go with **option 1 ONLY if user signs off**
— this unlocks real Phase 2 progress and the fork-execution story is the
biggest selling point for the project beyond "a better logger". Fallback
is option 2 (LangSmith extractor) as a crisp 1-round ship.

## Round 16 retrospective

What went well:
- Version-bump checklist caught the CLI status line drift cleanly (no
  fixture drift this round).
- `chronos info` smoke test caught nothing — confirms the bump landed
  right the first time.
- R13 pattern reusable: "pure release round" now a proven template.

What to carry forward:
- **When bundling multiple rounds into one release**, name the CHANGELOG
  section accordingly (e.g. `Round N + N+1 + N+2`). R13 did R12+R13;
  R16 did R14+R15+R16. Pattern works.
- **Milestone naming discipline**: not every version bump needs M bump.
  Reuse the M label when the work is a straight extension.

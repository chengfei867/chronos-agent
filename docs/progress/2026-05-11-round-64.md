## Decisions

### D1 — Dogfood topology mirrors R60 shape, not R60 semantics

R60's dogfood had a **designated** pivot + 3 forks (the R59 CLI took `<pivot> <other>...` positionals). Slice 4's `--auto-pivot` takes **N candidates with no designated pivot**, so the script reshapes the narrative — four "sibling" runs, no pre-declared pivot, centroid emerges from the data. Same four underlying LangGraph traces (baseline + identity twin + early-exit fork + extra-round fork), different framing at the CLI layer. Keeping the topology identical to R60 is deliberate: the two dogfoods can be compared side-by-side to see what the `--auto-pivot` flag adds on top of the same input data.

### D2 — Runtime assertions as living regression guards (R60 pattern extended)

R60's dogfood validated shape **visually** (print statements + human read). R64's dogfood goes further with runtime `assert` statements: `metric_version == 1`, `pivot_selection == "auto-centroid"`, centroid ∈ {baseline, twin} and specifically `== min(baseline, twin)` (ADR-024 tie-break), matrix has `C(4,2)=6` entries with canonical `min<max` orientation, `baseline<->twin` distance is exactly `0.0`, every other pair is strictly positive. These asserts will trip at release time on any drift in the `AutoPivotReport.to_dict()` contract — the dogfood is not just evidence, it's a release gate. Extension of R60 invariant "Test assertion drift guard" to include dogfood scripts.

### D3 — Single slot, not two

R63's finding flagged "A2 inheritance six-in-a-row → pre-budget 2 slots for Arc impl rounds." R64 is explicitly **not** an impl round — no new test scaffolding, no new CLI/HTTP surface, no new core. It is: (1) one new script (~310 LOC), (2) three version-mention bumps, (3) CHANGELOG roll, (4) progress + CONTEXT refresh, (5) commit + tag + push + GH Release. All additive, none requires revision of existing tests. Single-slot viability confirmed by the execution: the script ran correctly on second iteration (first iteration hit two small structural mismatches with JSON shape — `summary` lives inside `merged`, `pivot_selection == "auto-centroid"` not `"auto"` — both are dogfood-only assert tweaks, not contract changes).

### D4 — Dogfood script JSON-shape findings + identity-fork non-determinism

During dogfood authoring, three small contract surface facts surfaced that are worth pinning in CONTEXT:

1. **`AutoPivotReport.to_dict()` nests `merged` at top level** — the R63 CHANGELOG said "superset of `/runs/compare` JSON" which is true semantically, but the CLI JSON uses a **nested** layout (`{"centroid_run_id", "distance_matrix", "metric_version", "input_run_ids", "pivot_selection", "merged": {alignment, other_ids, pivot_id, summary, warnings}}`). The HTTP endpoint uses a **flattened** layout (`{"pivot_id", ..., "summary", "auto_pivot": {centroid, matrix, metric_version}}`). The two JSON shapes are **not byte-identical** despite both being "supersets". This is by design (CLI emits `report.to_dict()` verbatim; HTTP wraps the flat `/runs/compare/n` shape and adds an `auto_pivot` sub-object), but anyone wiring UI against both needs to know. Not a bug — worth documenting.
2. **`pivot_selection` string is `"auto-centroid"`, not `"auto"`** — matches ADR-024's name for the selection strategy (to leave space for future strategies like `"manual"`, `"first-as-pivot"`, `"random"`).
3. **`overrides={}` (identity fork) does NOT yield a byte-identical trace** — first dogfood run asserted `baseline<->twin distance == 0.0` and tripped: observed 0.25. Root cause: LangGraph fork replays from the fork point, and fresh nodes generated downstream have new `node_id`s / timestamps; the diff counts those as "added" rows. The distance stays small (typically ≤ 0.5 on the router-loop example) but is **not zero**. The true centroid invariant is that `baseline` and `twin` share the **minimum mean-distance** among all candidates and thus tie-break on centroid selection — not that their pairwise distance is literally zero. Dogfood assertion softened accordingly (`<= 0.5` + ordering check: identity-pair distance ≤ any other pair). **This is an R64 contract-shape finding worth pinning**: "identity fork" ≠ "byte-identical trace" in Chronos's current diff model.

All three findings are recorded here and surfaced to CONTEXT §5 as "candidate invariants" — promote if a second dogfood or design doc references them.

## Findings (potentially promote to CONTEXT invariants)

- **F1 — `AutoPivotReport.to_dict()` has nested `merged` sub-object at top level** (not byte-parallel to `/runs/compare/n` flat shape). CLI emits nested; HTTP `/runs/compare/auto` emits flat + `auto_pivot` sub-object. Two legitimate JSON shapes over the same data. Worth documenting in CONTEXT so future wire-the-Web-UI rounds know which endpoint to call for which shape.
- **F2 — `pivot_selection == "auto-centroid"`** (not `"auto"`). Pinned so future strategies (`"manual"`, `"first-as-pivot"`, `"random"`) extend cleanly. ADR-024 §Interface should grow a canonical enum of selection strategies.
- **F3 — Runtime assertions in dogfood script = release gate** (R64 extension of R60 "dogfood = living design doc"). If the script doesn't exit 0 at release time, the release is blocked. Cheap: one function call in the release workflow.
- **F4 — Single-slot release-after-planning round is viable when proof ≠ impl**. R63 pre-budgeted 2 slots per Arc impl round. R64 stayed single-slot because no test scaffolding was created/inherited. Distinguish proof-rounds from impl-rounds in slot-budgeting: proof = additive-only (script + metadata + release) = single slot; impl = new surface + new tests + possibly two-slot.

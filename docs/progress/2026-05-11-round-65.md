## R65 findings (candidate invariants)

### F1 — "Cross-endpoint argmin = centroid" locked as free third-layer Arc A guard (R65 new)

Prior to R65, the Arc A cross-layer guards were:
- R58 pure `merge_pivot_reports` ↔ pure `auto_pivot_compare`
- R59/R63 pure ↔ CLI ↔ HTTP `/runs/compare/n` ↔ HTTP `/runs/compare/auto` (N=2 byte-for-byte)

R65 adds a **semantic** (not byte-identical) cross-endpoint invariant that is cheaper but stronger in a different direction: `argmin(matrix.mean_distances) == auto.centroid_run_id` for any identical input set. This locks three layers of centroid selection — pure `select_centroid`, matrix endpoint's argmin, auto endpoint's centroid — to agree. If `select_centroid`'s tie-break rule ever drifts (say, a future refactor accidentally drops the lex tie-break), this test trips immediately from two different angles: the matrix endpoint argmin changes at the wrapper layer, the auto endpoint centroid changes at the core layer, and they diverge.

Locked by `test_compare_matrix_argmin_agrees_with_auto_pivot_centroid` (API test).

### F2 — "Surface mean_distances in matrix view, keep core merge-free" (R65 design note)

`select_centroid` needs `mean_distances` internally, but `pairwise_distances` intentionally does not return them — that would couple the pairwise pure function to the centroid-selection concern. R65 chose to compute `mean_distances` in the wrappers (CLI `_run_matrix` and HTTP `compare_runs_matrix`), not in core. Reasons:
1. `pairwise_distances` stays merge-free + reusable (e.g. a future 2D embedding visualization would want the matrix raw).
2. Wrapper-layer computation is O(N²) (same complexity as the matrix itself) — no asymptotic cost.
3. If a future consumer wants mean distances without the surfacing overhead, they can call `select_centroid(run_ids, pairwise_distances(...))` directly. This is the "wrappers push derived fields, core stays minimal" discipline.

Pin as design principle: **derived-but-cheap computations belong in wrappers, not core**.

### F3 — `seeded_compare_db` + `compare_n_scenario` fixtures are now cross-slice shared (R59 refinement 二度验证)

R63 refined R59's "new fixture new scenario" principle to "reuse unchanged fixture OK, only mutation forbidden". R65's tests reuse both `seeded_compare_db` (CLI) and `compare_n_scenario` + `compare_n_client` (HTTP) verbatim across slice 4 and slice 5 endpoints — second in-project validation that the refinement is correct. No fixture mutation; both slices exercise different endpoints against the same DB.

## Deliverables

### 1. `GET /runs/{run_id}/nodes/{node_id}/fork-plan` — new FastAPI endpoint

`src/chronos/api/server.py` (+46 lines):

```python
@app.get("/runs/{run_id}/nodes/{node_id}/fork-plan")
def get_fork_plan_preview(run_id: str, node_id: str) -> JSONResponse:
    ...
    plan = build_plan(parent_run=run, parent_node=parent_node,
                      overrides={}, child_thread_id=None, reason=None, tags=[])
    downstream = [n for n in nodes if n.step_index > parent_node.step_index]
    effects_summary = build_effects_summary(downstream)
    return JSONResponse({"plan": plan.to_dict(), "effects_summary": effects_summary})
```

- Reuses the **`build_plan` + `build_effects_summary` pure helpers** already exported from `chronos.cli.fork` (R45-A designed them to be console-free for exactly this purpose). Zero duplication.
- Empty `overrides={}` envelope — the UX is "download the JSON, edit overrides in your editor, then `chronos fork apply <plan.json>`". No in-browser editing (ADR-019 explicitly rejected that: any editor invites users to assume Chronos executes the fork, which it doesn't).
- 404 branches: "Run not found" and "Node not found" — distinct `detail` strings so the frontend can surface a useful message.

### 2. Frontend `fetchForkPlanPreview` in `frontend/src/api.ts`

Typed response envelope:

```ts
export interface ForkPlanPreviewResponse {
  plan: Record<string, unknown>;
  effects_summary: {
    total: number;
    dangerous_count: number;
    tag_counts: Record<string, number>;
    dangerous_samples: Array<[number, string, string[]]>;
  };
}
```

`plan` stays loosely typed as `Record<string, unknown>` — the UI just serializes it to JSON for download/copy, no structural destructuring. Keeps us decoupled from the `ForkPlan` Pydantic schema (R45-A still calls v1 "the v0.1.1+ contract" but the UI doesn't need to enforce shape).

### 3. `ForkPlanModal` component (`frontend/src/components/ForkPlanModal.tsx`, 252 lines)

AntD `Modal` (not `Drawer`) because the plan JSON is a **deliverable**, not a sidebar aux panel. 760px wide, footer has Close / Copy / Download buttons.

Layout top-to-bottom:
1. Intro paragraph — "Chronos plans, doesn't execute" reminder (ADR-013 in plain English).
2. Loading / error states.
3. **Effects summary panel**: either a `warning` Alert mirroring the CLI panel ("⚠ This fork re-runs N node(s) with real-world side effects", tag breakdown as Tags, up to 3 example nodes as bulleted list) — or a `success` Alert for the last-node/pure-LLM case.
4. Plan JSON block — dark-themed `<pre>` with `maxHeight: 320px` scroll (so 200-step runs don't blow out the modal).
5. Next-steps paragraph telling the user exactly what command to run.

**Silence principle** mirrored from R45-A: if `dangerous_count == 0` we still show a **success** Alert instead of nothing, because in a modal context silence is confusing (user clicked "Fork from here" expecting feedback). CLI can silently skip the panel; modal cannot.

### 4. TreeView wiring (`frontend/src/pages/TreeView.tsx`, +23 lines)

## Key facts / pitfalls surfaced

- **`build_plan` + `build_effects_summary` are callable from the API layer without any Rich Console import.** R45-A got this split right for exactly this reason — R46-A validates the separation. Note for future: anything that needs a `console.print()` (e.g. `render_effects_preview`) is NOT reusable; only the pure helpers are.
- **`scenario` fixture carries n2 at step_index=1 and n3 at step_index=2** — so forking at n2 has exactly 1 downstream node (n3). Locking this in the test makes it a canary if someone adds a 4th node to the fixture.
- **"Run not found" vs "Node not found" separation**: always check run existence *first* — otherwise a bad `run_id` + bad `node_id` surfaces as "Node not found" which is wrong (run wasn't there to begin with).
- **Modal-vs-Drawer for secondary content**: drawer = aux/passive (node metadata), modal = deliverable/action (fork plan). Followed R38/R39-A visual-review norms.
- **Cron slot handoff without progress doc = silent recovery burden**: this round inherited live uncommitted code with broken i18n. The frontend would have built clean and **shipped visibly-broken copy** to any user who ran `chronos web` between cron rounds. Lesson: always either commit WIP behind a feature flag, or stash/reset before ending a cron — never leave half-wired runtime dependencies in working tree. **Add a new CONTEXT §5 bullet for this.**
- **`gh-proxy.com` push still the only working route** (R22 fact, re-verified this round — see push log in git).

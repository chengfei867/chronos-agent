## 2. What landed

### 2.1 `tests/spikes/spike11_langgraph_tool_effects.py` (new, 260 lines)

An offline, no-LLM-needed spike that builds a 4-node
`StateGraph(State)` with function-shaped names intentionally chosen to
trip each effect-classifier group:

```
plan               → pure FN / router        (expected effects: [])
fetch_weather_api  → matches NETWORK_PATTERNS (expected: ["network"])
read_file          → matches FS_PATTERNS      (expected: ["fs"])
query_db           → matches DB_PATTERNS      (expected: ["db"])
```

The spike runs the same compiled graph twice through
`LangGraphRecorder`:

- **Run A** (no `kind_map`) — all nodes default to `NodeKind.FN`, so
  `classify_effects`'s `kind == NodeKind.TOOL` gate never fires, and
  every node gets `effects=[]`. Documents the real usage gotcha.
- **Run B** (`kind_map={"fetch_weather_api": TOOL, ...}`) — tool nodes
  produce the expected per-node effect tag.

Four findings, all ✅ on first run:

- **F1**: LangGraph `node_name` is single-segment and equals the
  `add_node(name, fn)` key (no prefix / suffix / colons).
- **F2**: Without `kind_map`, every node gets `effects=[]` — confirms
  the usage trap.
- **F3**: With `kind_map`, classifier picks the right tag per node
  (`["network"]` / `["fs"]` / `["db"]`).
- **F4**: ADR-020 vacuously satisfied. The invariant ("classifier input
  contains the tool function name") is met by LangGraph's default node
  naming — no three-segment surgery needed.

Ruff-clean, ruff-format-clean, not collected by pytest (lives in
`tests/spikes/` which is pytest-excluded per spike convention).

### 2.2 `docs/research/r49-langgraph-adr020-audit.md` (new, 184 lines)

The audit closure note. Documents:

- The spike's four findings in prose with the ADR-020 section
  cross-references.
- The **`kind_map` gotcha**: LangGraph has no framework-level concept of
  "tool node vs function node", so the adapter conservatively defaults
  everything to `NodeKind.FN`, and the classifier correctly refuses to
  guess. Users who want Phase 3 effect annotations must declare their
  tool nodes via `kind_map={"name": NodeKind.TOOL, ...}`.
- Consequence for Phase 3 UX: LangGraph users who don't know about
  `kind_map` will silently see green "no dangerous side effects"
  fork-modal Alerts even when their graph writes files / hits HTTP /
  queries DBs. **Silent false-safety is worse than a loud false-alarm**,
  which is why documentation is the remediation, not a code change.
- Recommendations:
  1. Add a one-paragraph note to `LangGraphRecorder.__init__` docstring
     telling users to pass `kind_map` if they want effect annotations
     (R50 candidate patch, bundled with the screenshot refresh).
  2. Amend ADR-020 follow-ups to mark LangGraph closed (done this
     round).
  3. No test suite addition needed — the classifier pipeline is
     already covered by `tests/unit/test_effects_classify.py` (R44-A)
     and `tests/integration/test_dual_adapter_dogfood.py`.
- Explicit **non-targets**: `langgraph.prebuilt.ToolNode` branches and
  subgraph-as-node behavior. Both are expected-but-unverified; neither
  is blocking R49 and neither is currently in any Chronos test path.

## 3. What was intentionally deferred

### 3.1 Screenshot refresh + `scripts/capture_fork_modal.py` helper → R50

CONTEXT.md §6 (R48-C) named this an R49 must-do. It is deferred again
for the same reason R48-C deferred it from R48-C itself: the wall-time
risk profile inside a cron slot is wrong.

- No Playwright is in `pyproject.toml` deps; installing it in-slot
  risks a 2-3 minute `uv sync` + browser download (~400 MB Chromium).
- The Vite dev server needs cold-build and bind-port negotiation.
- The hot-reload race between seed DB → API restart → dev-server
  re-fetch is well-known to be flaky on first attempt.
- R49 has one low-risk deliverable (the audit) and one high-risk
  deliverable (the screenshots). Shipping the audit alone is strictly
  better than failing to ship both.

**R50 plan for the screenshots** (documented so next round's agent
doesn't re-litigate):

1. Add `playwright` to `[project.optional-dependencies.dev]` or a new
   `[project.optional-dependencies.screenshots]` group.
2. Write `scripts/capture_fork_modal.py` that:
   - Opens `SqliteStore.open(dogfood.db)` and runs
     `scripts/seed_r47a_effects.py` if the three runs are absent.
   - Launches `chronos web --no-browser --db dogfood.db` in a
     subprocess.
   - Launches `cd frontend && npm run dev` in a subprocess (or uses
     the pre-built `frontend/dist/` via `CHRONOS_FRONTEND_DIST`).
   - Uses Playwright async API to open three pages (`#/runs/<warn>`,
     `#/runs/<safe>`, `#/runs/<lastnode>`), click the fork-plan
     CTA, wait for the modal, save the three PNGs to
     `docs/images/fork-modal/{01,02,03}-<suffix>.png`.
3. Document the script in `docs/guides/screenshot-capture.md` (new).
4. Run the script, commit the three new PNGs, update
   `docs/forking-safely.md` image references if the filenames
   changed.

Work budget estimate: 1 full R50 slot.

## 5. Decisions

### D-49-1: Prioritise the audit over the screenshots

Already litigated in §3.1. Short form: shipping one deterministic,
self-contained deliverable beats attempting two and potentially
shipping zero. The audit closes a real open ADR follow-up that's been
pending since R48-A. The screenshots are cosmetic.

### D-49-2: Document the `kind_map` gotcha in the research note, not in source docstrings (this round)

The research note is the right home for the full "why" and the
recommended remediation wording. The docstring patch is the right
home for the "do this" one-liner. They're decoupled deliverables and
the docstring patch pairs better with the screenshot refresh than
with the spike.

### D-49-3: CrewAI follow-up stays open in ADR-020

Writing a CrewAI adapter is a 3-5-round effort (ADR-021 + scaffold +
first real run + effect annotations firing). R49 is not the round for
it. Leaving ADR-020's CrewAI follow-up explicitly open — rather than
removing the whole Follow-ups section — keeps the future audit
prompted.

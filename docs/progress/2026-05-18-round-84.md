## §1. Strategy decisions

### §1.1. Two divergent stub shapes — refactor 5, defer 1

A grep audit of `_StubBlock` / `_StubMessage` / `_aiter` references surfaced **6 sites** (CONTEXT.md §6 R84 hand-off matched), but two **structurally different** shapes:

| Shape | Files | Style |
|---|---|---|
| **Static-subclass** (refactored) | `test_anthropic_agents_fork_tool_override.py`, `test_anthropic_agents_fork_tool_result_override.py`, `test_queries_tool_linkage.py`, `dogfood_fork_tool_override.py`, `dogfood_fork_tool_result_override.py` | `_StubBlock` base + named subclasses `TextBlock` / `ToolUseBlock` / `ToolResultBlock` and `UserMessage` / `AssistantMessage`. Identical fields across all 5. |
| **Runtime-factory** (deferred) | `test_adapter_anthropic_agents.py` | `_StubBlockBase` with **richer fields** (`is_error`, `thinking`, `signature`) + `_blk(cls_name, **kw)` runtime-subclass factory. No static subclasses. |

Refactor scope: **5 sites migrated**, 1 site **intentionally deferred**. Merging would either bloat the shared module's static surface (carrying `is_error`/`thinking`/`signature` for callers that don't need them — R64 anti-pattern: future-proof slots are falsification targets) or drop the named-subclass ergonomics 5 of 6 sites depend on. R85+ may reconcile if a forcing function appears.

The shared module exports both forms (`StubBlock` static + `make_block(cls_name, **kw)` factory) so the deferred file *could* migrate later without growing the module's surface.

### §1.2. Module location: `tests/unit/fixtures/` (existing convention) not `tests/fixtures/` (CONTEXT.md §6 hint)

R58 created `tests/unit/fixtures/three_run_pivot.py` and `tests/unit/test_merge_pivot.py:29` already imports `from tests.unit.fixtures.three_run_pivot import (...)`. The CONTEXT.md §6 R84 hint suggested `tests/fixtures/` (sibling to `tests/unit/`), but adopting two parallel fixture modules would dilute the R58 convention. **Decision: use `tests/unit/fixtures/anthropic_agents_stubs.py`**, keeping the project to one fixture root.

### §1.3. Dogfood `sys.path` bootstrap (4 lines)

Pytest adds the repo root to `sys.path` automatically (because `[tool.pytest.ini_options].testpaths = ["tests"]`), so `from tests.unit.fixtures.* import ...` resolves naturally inside tests. Dogfoods, however, run as `python scripts/dogfood_*.py` and only have `''` (cwd) + the installed-package's `src/`-prefixed entry on `sys.path`. The shared module is **not** distributed inside the package (it's a test-only helper).

Two viable patches:

* **A.** Move the module under `src/chronos/_testing/` so it ships with the package. ✗ Pollutes the public install with test fixtures; warrants an ADR.
* **B.** Bootstrap `sys.path` in each dogfood (4 lines: import `Path` + `sys`, `sys.path.insert(0, repo_root)`). ✓ Already-imported modules; localized; honest about what's happening.

**Chose B.** The bootstrap block is wrapped in a comment explaining why and pointing at R84.

## §3. Shared-module API

`tests/unit/fixtures/anthropic_agents_stubs.py` exports:

```python
__all__ = [
    "AssistantMessage",
    "StubBlock",
    "StubMessage",
    "TextBlock",
    "ToolResultBlock",
    "ToolUseBlock",
    "UserMessage",
    "aiter_messages",
    "make_block",
    "make_message",
]
```

### Static taxonomy (used by 5 refactored sites)

* `StubBlock` — `@dataclass` with optional `text` / `name` / `input` / `id` / `tool_use_id` / `content` slots. Recorder dispatches kind on `type(block).__name__`, so subclass names matter, isinstance does not.
* `TextBlock`, `ToolUseBlock`, `ToolResultBlock` — empty subclasses of `StubBlock`.
* `StubMessage` — `@dataclass` with `content` / `usage` / `model` / `stop_reason` / `total_cost_usd` / `duration_ms` / `uuid` / `session_id` / `extra` slots.
* `UserMessage`, `AssistantMessage` — empty subclasses of `StubMessage`.
* `aiter_messages(messages: list[Any]) -> AsyncIterator[Any]` — drop-in for the previous `_aiter`.

### Runtime factories (parity with `test_adapter_anthropic_agents.py` style; NOT yet consumed)

* `make_block(cls_name: str, **kw) -> StubBlock` — runtime subclass with the given class name (for ad-hoc kinds without polluting the static taxonomy, e.g. `make_block("ThinkingBlock", text="...")`).
* `make_message(cls_name: str, **kw) -> StubMessage` — companion for messages.

The factories are in place so a future R85+ migration of `test_adapter_anthropic_agents.py` is mechanical, not a redesign. Their inclusion is *defensive*, not *speculative*: the deferred file already calls `_blk(cls_name, **kw)` and `_msg(cls_name, **kw)` 30+ times, so the API is anchored in real existing usage, not future hypothetical needs (R64 anti-pattern check passed).

### Non-goals

* **Does NOT replace `test_adapter_anthropic_agents.py`'s `_StubBlockBase`.** That file uses richer fields (`is_error`, `thinking`, `signature`); merging would either bloat or break asymmetric.
* **Does NOT export `_FakeClient`.** Each call site has slightly different `connect()` / `disconnect()` / awaitable shapes; the helper isn't a clean shared API.
* **Does NOT pre-add `is_error` / `thinking` / `signature` slots to `StubBlock`.** R64 invariant: future-proof slots are falsification targets. Add when a consumer needs them, not earlier.

## §4. Refactor mechanics (the part that surprised me)

The static-subclass form has stable internal references — every test references `_StubBlock` / `_StubMessage` / `_aiter` 5–12× per file. Four-step recipe per file:

1. **Rename**: regex sub `\b_StubBlock\b → StubBlock`, `\b_StubMessage\b → StubMessage`, `\b_aiter\b → aiter_messages` across the whole file.
2. **Cut**: remove the stub-def block (from `# --- Stub ...` section header to right before `@dataclass\nclass _FakeClient`). For `test_queries_tool_linkage.py`, end-marker is `# Fixtures` instead because it has no `_FakeClient`.
3. **Import**: insert the fixture import after `import pytest` (or for dogfoods, after the chronos imports + sys.path bootstrap).
4. **Auto-fix unused imports**: `ruff check --fix` clears the F401s that the over-broad import block introduced (e.g. `test_queries_tool_linkage.py` doesn't use `StubMessage` directly — only the subclass `UserMessage` / `AssistantMessage`).

Ruff's auto-fix removed 13 unused-imports across the 4 import blocks — let it shrink to per-file actual usage. Cleaner than hand-tailoring 4 different import blocks. **Pin: prefer over-broad import + ruff auto-fix to per-file hand-tailored imports** (anti-bikeshed).

## §5. Findings on the wall

### F1 — *Static-subclass vs runtime-factory is a real shape distinction* (R84 new)

Six call sites looked uniformly duplicated, but a 5-min audit surfaces two structurally distinct stub shapes. The cheap-to-extract one (5 sites, identical) is worth doing; the expensive one (1 site, divergent) is worth deferring with a defensive factory in place. **Pin: when duplication looks N-fold, audit shape divergence before extraction; partial extraction with deferred reconciliation is a valid outcome.** Compounds the R66 retro-documentation finding ("audit before re-build"): **audit before refactor** is the same principle one level down.

### F2 — *Dogfood `sys.path` bootstrap is acceptable boilerplate when the alternative is shipping test helpers in the public package* (R84 new)

4 lines of bootstrap is a small price compared to either (a) creating `src/chronos/_testing/` (architecturally invasive, ADR-deserving) or (b) duplicating the fixture in `scripts/_testing_helpers/` (defeats the refactor). **Pin: dogfoods are entry-points, not library code; explicit `sys.path.insert(0, repo_root)` is the most honest seam between "installed package can find" and "repo-checkout-only helpers".**

### F3 — *Over-broad import + ruff auto-fix beats hand-tailored per-file imports* (R84 new)

4 files imported the full 8-symbol surface; ruff `--fix` removed 13 unused imports across the 4 sites in one shot. Hand-tailoring would have been 4 mental allocations and 4× rebase risk if the fixture surface evolves. **Pin: when refactoring N call sites toward a shared module, import everything; let ruff trim per-file. Lower friction, lower drift.**

### F4 — *The R83-listed "first-choice R84 cleanup slot" really was a single-slot job* (R84 confirmation)

R83 progress doc + ADR-026 §6 retro + CHANGELOG R83 placeholder all converged on "extract fixture, single slot, low risk". Wall-clock: ~25 min from `git status` (cron slot start) to commit-ready, plus ~5 min for this progress doc. R83's read on the slot budget was accurate.

### F5 — *R83's CHANGELOG `Known caveats` line "duplicated across 3 unit-test files" was an UNDER-count* (R84 retro)

R83 wrote: "stub patterns (`_StubBlock`, `_StubMessage`, `_aiter`) duplicated across 3 unit-test files". Actual count: **6 sites** (3 unit + 2 dogfood + 1 differently-shaped unit). The under-count came from focusing on the immediately-relevant slice-3 files; R84 audit caught the fuller picture. Not a correction worth fixing in CHANGELOG R83 entry post-tag, but pin the lesson: **caveat counts in release notes should come from a grep, not from memory.**

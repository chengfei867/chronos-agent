# Round 48-A — AutoGen effects classifier fix + ADR-020

**Date**: 2026-04-27 (CST, ~01:30)
**Round**: R48-A
**Theme**: Close the Phase 3 blind spot — effects classifier was broken on AutoGen tool events for 4 rounds (v0.3.0 → v0.4.0a1)

## TL;DR

Phase 3's fork-safety pipeline (ADR-019 + effects classifier + Web modal warning) *looked* like it worked across all adapters because every unit test passed. R48-A ran the first **real LLM × real AutoGen tool-calling** spike (spike 10) and discovered the classifier was emitting `effects=[]` on **every single AutoGen tool node** — the Web UI warning pipeline was silently blind. Root cause: the recorder synthesized `node_name = "{source}:{ClassName}"` (e.g. `"coder:ToolCallExecutionEvent"`) with zero tool-function-name signal, and the classifier's keyword regexes only match function-shaped strings.

Fix: embed the `FunctionCall.name` from `msg.content[*].name` as a third segment in `node_name`. `{source}:{EventClass}:{tool_name[+tool_name...]}`. Classifier now fires. 4 new unit tests, 1 spike, 1 ADR (ADR-020 codifying the convention for future message-based adapters), + `forking-safely.md` §6 on per-tool `effects_map` overrides.

This was inherited WIP from a prior cron slot that did the spike + research + recorder fix + test scaffolding, left it uncommitted. This round (a) applied lint/format fixes (SIM108 + RUF001), (b) wrote ADR-020, (c) wrote the guide §6, (d) wrote this progress doc, (e) committed + pushed.

## Observations

### Why the bug shipped for 4 rounds

R44-A's classifier tests used synthetic `node_name` strings that were already function-shaped (`"fetch_weather"`, `"read_file"`). No test ever exercised the **actual AutoGen-adapter → classifier** data path. The stack went green end-to-end without ever proving the pipeline worked on real adapter output.

LangGraph was accidentally fine because graph-level node names come from `graph.add_node("fetch_weather", ...)` — already function-shaped by construction. AutoGen, being message-based, has no natural function-shaped identifier per node; the recorder had to synthesize one and **chose** a shape that lost the tool name.

Lesson (added to ADR-020): "every classifier test with `NodeKind.TOOL` should either (a) use a name we've empirically confirmed the adapter produces, or (b) feed through the real recorder path."

### The three-segment convention

Decided (ADR-020) that all message-based adapters should synthesize:

```
{source_or_agent}:{Kind_or_ClassName}:{tool_name[+tool_name...]}
```

Rationale: it's cheap (~15 lines per adapter), it preserves the existing single-string `effects_map` contract, and it gives users per-tool × per-event × per-agent override granularity. LangGraph and other graph-based adapters are exempt — the invariant is "classifier's input contains the tool function name", not "exactly three segments".

### Soft-break on `effects_map` keys

Users with old-shape keys (`"coder:ToolCallExecutionEvent"`) will see their overrides silently no-op after upgrade. Acceptable because (a) effects on AutoGen *didn't actually work* before this fix, so very few users had production overrides, (b) CHANGELOG calls it out, (c) discovery path is obvious once a user inspects `nodes[i].node_name`.

## Decisions

1. **Commit inherited WIP + new artifacts as one logical unit (Option A from the handoff skill).** Don't fragment into "lint fix" + "ADR" + "guide" commits — the story is "close the Phase 3 AutoGen blind spot" and it lands as one thing.
2. **`node_name` shape change does NOT trigger a major version bump.** `Node.node_name` is documented as adapter-internal; we've never promised stability. The soft-break is on `effects_map` keys, which is documented in the CHANGELOG as expected from an alpha → alpha transition.
3. **ADR-020 separate from ADR-019.** ADR-019 is charter-level ("Chronos does not sandbox"). ADR-020 is a convention for a specific mechanism (how adapters must shape `node_name` so the classifier works). They live at different altitudes.
4. **No release this round.** v0.4.0a2 is the natural next alpha, bundling this fix with whatever R48-B / R48-C produce (badge redesign, i18n hardening). A single-fix release would duplicate the R47 mistake of too-small releases.
5. **Spike 10 stays as a spike**, not promoted to a `@pytest.mark.live` test this round. Research doc's Follow-ups says "consider promoting next round if OneAPI/GLM-5 remains stable" — one round of stability isn't enough yet.

## Lessons (CONTEXT.md §5 new bullet candidates)

- **Classifier tests MUST use real adapter output, not synthetic strings.** If you're testing a function that reads `node.x` and `x` is produced by adapter code, your test fixture's `x` must come from exercising the adapter. Otherwise you validate the function, not the integration.
- **`cron-slot-handoff-recovery` skill Option A works well when the prior slot did the hard thinking.** Spike + research + code was all there; this round's job was to polish (lint), codify (ADR), document (guide §6), narrate (progress doc), ship. Total: ~45 minutes of agent time, no wasted work.
- **Keyword-regex classifiers are fragile to input-shape drift.** When we add CrewAI / AG2, the first test must be "feed an actual adapter-produced `node_name` into the classifier and verify it fires". Not "unit test the classifier with a hand-chosen string".

## §3 — Decisions / rationale

### D-1: Option (a) over option (b)

R85 noted two possible reconciliation paths:

- **(a)** Document `kind=LLM` for tool blocks as intentional (envelope-determines-kind).
- **(b)** Split blocks into separate nodes via ADR-027.

Chose **(a)** because:

1. **Post-GA**: v0.7.0 shipped 2026-05-19 with the envelope behaviour. Splitting per-block would be a breaking schema change — node count for any tool-using run would multiply, breaking diff invariants, replay storage, and external tooling assumptions.
2. **Code already canonical**: 36-round zero-regression streak across LangGraph, AutoGen, CrewAI, anthropic_agents — the envelope rule has held silently for the entire project history. Codifying it isn't a *new* decision; it's surfacing what's already true.
3. **Cost asymmetry**: (a) is 1-round md-only. (b) would be: ADR-027 + per-block split refactor in recorder.py + R76/R77 `tool_use_id`/`tool_use_ids` re-stamp logic + 30+ test rewrites + dogfood updates + alpha→GA cycle for v0.8.0. (a) costs ~$0.01; (b) costs 6+ rounds.
4. **Mental model**: "one Node = one observable agent step" maps naturally onto "one Node = one SDK message". Per-block split would break this.

### D-2: Keep dead map entries, document them

Pruning `ToolUseBlock`/`ToolResultBlock` from `_DEFAULT_KIND_MAP` would be the orthogonal "code cleanup" round. Decided against because:

- They are **harmless** (never reached at runtime under the SDK's current top-level iterator contract).
- They are **forward-compatible** — if a future SDK release surfaces blocks as top-level stream items, the entries activate without code change.
- They are **defensively useful** — a custom `kind_map` callsite that wraps `_kind_for()` to defer per-block can fall through to these entries.
- Removing them would add a `git blame` entry that's pure rearrangement; not worth the noise.

### D-3: No `uv run pytest` this round

Docs-only changes touch no Python source / tests. Re-running the suite would (a) regen `uv.lock` if the env had drifted (lockfile-trap per R70), (b) cost ~17–18s wall + LLM overhead, (c) produce identical 648/9/0/0 baseline. Skipped per the cron-slot-handoff-recovery skill's "minimal-change recipe" guidance: gates verification is needed when src/ or tests/ changed; for md-only, the previous round's gate evidence is the baseline. The drift sweep `grep` self-checks (CHANGELOG header, ADR cross-ref) were run inline.

## §4 — Findings

### F-1: Doc drift can persist GA-long if not actively swept

The `UserMessage → fn` table entry has been wrong since R71 (2026-05-13) — through 18 rounds, 5 alpha cuts, and 1 GA. Detection vector this round was *only* the act of writing a contract doc that referenced the same table. **Lesson**: when documenting a contract, always cross-check against runtime source. ADR-026 had the right wording in its AC-2 closing note (R85 finding) but the per-adapter doc wasn't re-swept after the finding landed. Codified into the round as a side-fix rather than deferred.

### F-2: "Tracked for a future round" debt has a half-life

ADR-026 §6 AC-2's "tracked for a future round to either reconcile or document explicitly" phrasing was added 2026-05-18 (R85). It survived: R86 (release-engineering attempt + revert), R87 (GA cut), R88 (GA recovery), and R89-eve. **4-round half-life is the realistic ceiling for "tracked for future" inline ADR debt** — beyond that, the original author's context is gone and the next round either resolves it or normalises away the reminder. R89 resolved before the 5th-round drift point. **Codify as an invariant**: `cron-slot-handoff-recovery` skill should explicitly include "scan most recent ADR closing-notes for 'tracked for future' / 'TBD' / 'TODO' phrases and surface as round-candidate" in its diagnostic.

### F-3: Docs-only rounds are the right cadence-restorer after release-engineering churn

R86/R87/R88 were all release-engineering-flavoured (dogfood writing, GA cut, recovery). R89 deliberately picked the lightest contract-debt option to re-establish boring-round rhythm before the next implementation push (Phase 5 Arc selection or Option β offline-fixture AC-3). **Pattern**: after 2+ consecutive release-engineering rounds, schedule a docs-only round to drain accumulated md-debt. Adds to the "round cadence" wisdom alongside R64's "proof-round vs impl-round" budgeting.

### F-4: R48-B stale-ref trap re-confirmed at R89

`git status` initially reported `[ahead 2]` despite the round following a clean R88 commit. `git fetch origin main` advanced the local ref and dissolved the apparent ahead-state. Slot-N+1's local `origin/main` ref does NOT auto-update from slot-N's push; this remains structurally invariant. **The skill already documents this** (`cron-slot-handoff-recovery` Step 1) — R89 is the Nth in-the-wild confirmation.

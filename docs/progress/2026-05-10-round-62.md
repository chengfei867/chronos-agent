### 3. Module implementation (SOP step 4b)

Laid out `src/chronos/core/auto_pivot.py` as **four tiers** so each step of ADR-024's algorithm is independently testable:

```
compute_distance(DiffReport) -> float                 # tier 1: one pair
pairwise_distances_from_reports(reports)              # tier 2: N(N-1)/2 pairs → matrix
select_centroid(run_ids, distance_matrix) -> str      # tier 3: argmin with tie-break
auto_pivot_compare(store, run_ids, ...) -> AutoPivotReport   # tier 4: orchestrator
```

**Distance formula** (ADR-024 §Algorithm): `d = (changed + added + removed) / total_rows`, with `total_rows = equal + changed + added + removed`, and `d = 0.0` when `total_rows == 0`. Symmetric (`d(a,b) == d(b,a)`) because `diff_runs` produces symmetric summaries for `equal`, while `changed + added + removed` is the multiset complement and is stable under swap.

**Matrix orientation**: canonical `(min_id, max_id)` keys, asserted in a helper `_canonical_pair(a, b) -> (str, str)`. Every `pairwise_distances_from_reports` test verifies the matrix only contains canonical keys — no accidental duplicates like `(A,B)` **and** `(B,A)` ever reach the centroid picker.

**Centroid rule**: `mean_distance_from(i) = sum_{j≠i} d(i,j) / (N-1)`; pick `argmin_i`, ties broken by lexicographic `min(tied_ids)`. Tested with an equidistant triangle: `{A:0.3, B:0.3, C:0.3}` → centroid = `"A"`.

**Orchestrator**: duck-types the store via a local `_AutoPivotStore` protocol (R15 — never import the concrete SQLite class). Accepts either a live `store` **or** pre-built `pairwise_reports` as an inject-seam — the latter is what every tier-3-and-below test uses, which keeps unit tests out of the store. For the tier-4 orchestrator tests we either pass `pairwise_reports=...` or use a real `SqliteStore` from `three_run_pivot_scenario`.

**The centroid re-diff step**: `pairwise_distances_from_reports` keys reports in canonical `(min_id, max_id)` orientation, so half of them have `run_a ≠ centroid` when the centroid isn't the lexicographically-smallest id. `merge_pivot_reports()` requires `run_a == pivot` for every input report (R58 invariant). Solution: after centroid selection, re-call `diff_runs(centroid, other)` for each non-centroid `other` — this re-orients every report with `run_a = centroid`. Cost: `N-1` extra diff calls; total stays `O(N²)` and is dominated by step 2's `N(N-1)/2` pairwise calls.

**`AutoPivotReport` dataclass**: `{centroid_run_id, distance_matrix, merged_alignment, warnings}`. `merged_alignment` is an R58 `MergedPivotAlignment` — byte-for-byte the same type every other Arc A consumer already speaks. No new JSON contract was introduced this round (deliberate — R63 will decide if `AutoPivotReport.to_dict()` adds `centroid_run_id` + `distance_matrix` as top-level keys or wraps `merged_alignment` under a sub-object).

**Warnings**: `N > 8` emits the same soft-cap warning ADR-024 and the design doc §7.1 specified for `chronos compare`.

## Deviation from ADR-024 (§6 hard constraint says direction drift must be argued in progress doc)

**ADR-024 §Interface specified**: `src/chronos/core/diff/auto_pivot.py` — i.e., `core.diff` would become a package with `auto_pivot.py` as a submodule.

**What I shipped**: `src/chronos/core/auto_pivot.py` — a **sibling** of `core/diff.py`, not a submodule.

**Why**:

1. **`core/diff.py` is a single 594-line module**, not a package. Converting it to a package (`core/diff/__init__.py` + moving its contents) is a cross-cutting refactor — `diff.py` currently hosts `DiffReport`, `MergedPivotAlignment`, `diff_runs`, `merge_pivot_reports`, plus internal helpers. Every import site (`src/chronos/cli/diff.py`, `src/chronos/cli/compare.py`, `src/chronos/api/server.py`, 8+ test files) currently says `from chronos.core.diff import ...`; if `diff.py` becomes `diff/__init__.py`, those imports still work, but any internal cross-import inside the package needs redirection.
2. **Risk-vs-return doesn't add up for a leaf-function add.** The blast radius of a package refactor includes the v0.5-frozen public contract (`DiffReport.to_dict`, `MergedPivotAlignment.to_dict`, `diff_runs`, `merge_pivot_reports`) plus the R58 invariants. A single slipped import or a subtle `__init__.py` re-export omission regresses the v0.5 contract. R62's goal is to ship the algorithm, not to reshape `core.diff`'s packaging.
3. **Intent preservation**. ADR-024's algorithmic substance — the distance formula, the centroid rule, the lexicographic tie-break, the `O(N²)` bound, the delegation to the R58 frozen `merge_pivot_reports` — is preserved byte-for-byte. Only the **import path** changes: `from chronos.core.auto_pivot import auto_pivot_compare` instead of `from chronos.core.diff.auto_pivot import auto_pivot_compare`.
4. **Forward compatibility**. When R63 lands the CLI/HTTP wrappers and (maybe) the package refactor, `core/diff/__init__.py` can re-export the `auto_pivot` names, and the current sibling module can be moved into the package with zero caller changes.

I will **not** backfill an ADR-024 amendment unless R63 surfaces a concrete case where the package layout was necessary. The roadmap and ADR-024 acceptance bar are met — the function exists, is correct, and is tested.

## 3. Findings ratcheted onto the wall

### F1. Aspirational-release-doc is a new trap class — codified into `cron-slot-handoff-recovery` skill

The trap shape:
- CHANGELOG `[X.Y.Z]` block freshly written with full Added/Changed/Quality bar + cost numbers like `~$0.10`
- ADR ACs flipped `[x]` claiming GA-gate green
- `[Unreleased]` empty / drained
- BUT `pyproject.toml` version unbumped, no progress doc, no tag, no commit

Inverse of release-mid-flight A2: **release-mid-flight A2 = real work shipped, ceremonial commit/tag pending; aspirational = ceremonial doc shipped, real work never observed**. Both look identical on disk. The distinguisher: the dogfood/live-smoke that the release block claims passed. Rerun it. If green, you're in A2 path. If exit 2 (relay degraded), defer + revert. If exit 3 (hard), the claim was fiction; revert.

Skill section added: \"Aspirational-release-doc trap (R86 lesson)\" with the recovery recipe (5 steps), the honesty rules, and the relay-flake corollary.

### F2. Relay-flake corollary — don't un-ratchet a recorded green AC just because the live run is currently red

R85 closed AC-2 with `~$0.14, INVARIANTS-GREEN` recorded. R86 today: same R85 dogfood **also** exits 2. Adapter code unchanged in 1 day. **The R85 ratchet is not retroactively invalid** — it really did pass at R85. AC-2 stays `[x]` on R85's recorded fact. AC-3 (claimed by prior R86 slot) goes `[x]` → `[~]` because it was never actually run green against the relay; the prior slot's ADR flip was prospective, not observational.

Generalization: a relay-coupled GA-gate AC's `[x]` ratchet is a **time-stamped snapshot** of \"this round, against this relay state, with this SDK pin, the dogfood asserted invariants and exited 0\". It is **not** a perpetual claim about the relay's future state. Relay degradations are environmental and don't unratchet adapter-code claims.

This is the right behavior because: (a) relays are external dependencies whose health is outside chronos's control; (b) re-running every prior green AC on every round would be cost-prohibitive and add no signal beyond the original ratchet; (c) treating \"can't currently demonstrate green\" identically to \"never could have been green\" punishes the project for environmental flakes.

### F3. SDK version updates change the relay-error envelope; classifier must keep up

R69 first documented the OneAPI relay's synthetic-auth-failed mode (text contains `\"authentication\"` / `\"synthetic\"` / `\"not logged in\"`).

R85 hard-coded those substrings into `arc_b_slice_3_mcp.py:266` as a 3-marker heuristic.

R86 hit **the same root cause** with **a different surface**: today's `claude-agent-sdk` (≥ R85's pin) wraps the relay-side `is_error=True ResultMessage(subtype='success')` into the misleading exception `Exception('Claude Code returned an error result: success')`. Neither `\"authentication\"` nor `\"synthetic\"` is in that string. R85's heuristic mis-classifies it as exit 3 (hard regression) — which is the exact failure mode that pushed prior R86 slot toward the aspirational doc (\"oh dear, looks hard, but maybe it's the relay anyway, just write the close-out and move on\" → revert).

Fix: extract classifier to `scripts/dogfood/_degradation.py`, broaden marker list to 4 substrings (add `claude code returned an error result`), and pin the heuristic with a parametrized unit test (`tests/unit/test_dogfood_degradation.py`) that grows monotonically as new relay-failure shapes appear. Future SDK version → new envelope → add 1 marker + 1 test case = belt-and-suspenders protection.

### F4. AC-3 has a deeper problem than R86's specific flake — it's relay-coupled

The general issue: AC-3 (\"real-relay override-fork live-smoke green\") **requires** a working OneAPI relay to demonstrate. As long as that's the only path, every relay flake blocks GA. R87+ should consider:

- **Option α**: keep waiting for relay recovery, accept that GA cut date is non-deterministic.
- **Option β**: author a deterministic offline-fixture-based AC-3 closure path. Pre-record a real session-protocol JSONL transcript (one-time, requires a green relay window), then replay against a local fake that emits the SDK's `Message` types from that JSONL. AC-3 becomes a fixture-driven assertion that the recorder + fork primitive correctly handle the SDK's actual session-protocol shapes, without depending on live relay health every release.

Option β is the **right long-term move**: GA-gate ACs should not be coupled to environmental dependencies the project doesn't control. R87 will pick this up.

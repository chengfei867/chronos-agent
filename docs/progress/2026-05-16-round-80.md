## What landed (close-out additions)

### 1. Dogfood script bugfixes (`scripts/dogfood_fork_tool_override.py`)

The prior slot wrote 326 lines of dogfood that exercised all 4 paths
end-to-end, but the fake `claude_agent_sdk.fork_session` and the
post-fork ref attribute access didn't match the recorder's actual
contracts. Three runtime bugs fixed in this slot:

1. **Fake `fork_session` signature mismatch** — recorder calls it as
   `fork_session(parent_session_id, up_to_message_id=..., title=...)`
   (positional first arg + kwargs); fake had `*, session_id: str,
   **kwargs` (keyword-only). Fixed to
   `def fork_session(session_id: str, /, **kwargs: Any) -> _FakeForkResult`,
   mirroring the unit-test fixture in
   `tests/unit/test_anthropic_agents_fork_tool_override.py:161-171`.
2. **Fake return type** — recorder reads `fork_result.session_id`; fake
   returned a bare `str`. Now returns a `_FakeForkResult` dataclass
   with a `session_id: str` field, again mirroring the test fixture.
3. **`ref.run_id` vs `ref.child_run_id`** — `recorder.record()` returns
   a `RunRef` with `.run_id`; `recorder.fork()` returns a `ForkRef` with
   `.child_run_id`. Three of the script's accesses had the wrong
   attribute. Plus a name-shadowing bug at the orphan-anchor lookup
   (`orphan_anchor` rebound from `str` to `Node`, then used as dict key
   on a later `tool_input_overrides={...}` line). Fixed by introducing
   a separate `orphan_anchor_node` binding.
4. **Over-narrow node filter** — `n.kind.startswith("AssistantMessage")`
   was filtering the override-stamp lookup, but the test harness in
   the unit suite doesn't filter by `kind` (just by
   `state_after['tool_use_id']`). Removed the redundant filter.

## Decisions / notes

- **Why `_FakeForkResult` over `SimpleNamespace`?** The unit-test fixture
  uses a frozen-style dataclass with a single `session_id` field;
  copying that exact shape keeps the dogfood script's contract identical
  to what the test suite blesses. Future SDK shape extensions (e.g.
  `parent_session_id`, `created_at`) will land in both places at once.
- **Why not collapse R79 + R80 into one commit?** R79 was already a
  separate logical unit (TDD scaffold + ADR amendment + no-op
  pass-through, all reviewable independently from impl). Two commits
  preserve the strict-xfail forcing function story: R79 lays the trap,
  R80 springs it. This slot ships **R80 only** because R79 was committed
  (or staged) by the prior slot in this same session.
  → Verified at start of slot: `git log --oneline -5` shows last commit
  is **R78** (fed0412). So R79 was *not* committed by the prior slot
  either — both R79 and R80 changes are uncommitted. **Decision**: pack
  both into a single R80 commit since they're inseparable now (xfail
  markers don't exist as a stable intermediate state — they were
  removed at impl-land time, as the strict-xfail design intended).
  CHANGELOG entries remain separated by round for readability.
- **Why `str(tid)` instead of an `assert isinstance(tid, str)`?** The
  surrounding 364-380 `multi-block` arm already filters on
  `isinstance(..., str) and id`; the `get()` lookup is downstream. Using
  `str(tid)` is a cosmetic narrow that satisfies mypy without inserting
  a runtime branch on a path we know is `str`. If future refactors
  break that invariant, mypy will catch it elsewhere.

## TL;DR

Two breakthroughs this round, in order of importance:

1. **R69 spike #1 prediction REFUTED.** R69 wrote that the OneAPI relay would not be `claude-agent-sdk` compatible because the SDK speaks a "session protocol" (init → text → result) that no relay would surface. R72 honoured that by writing the alpha gate as "live-smoke gated on real Anthropic auth, indefinitely." **R73 ran the actual probe and found the SDK round-trips cleanly through OneAPI** when the model id is given in baidu-int's spaced PascalCase form (`"Claude Sonnet 4.6"`) instead of the kebab-case `claude-sonnet-4-5` the SDK doc-strings show. The relay does forward the session protocol; R69's failure was a model-name-format issue, not a protocol issue.
2. **Arc B slice 1 live-smoke green end-to-end.** Three-tier dogfood probe (`scripts/dogfood/arc_b_slice_1_smoke.py`) and the two pytest live tests (`tests/live/test_anthropic_agents_smoke.py`) all pass against the real OneAPI Anthropic endpoint. Recorder produces three nodes (FN + LLM + END) per single-turn `query()`. The alpha release gate that R72 froze is now open.

Consequence: **v0.7.0a1 is cut and tagged in this round** rather than waiting on a future "real Anthropic auth" event that R72 thought was uncertain.

## What we learned (the spike refutation)

R69 spike #1 reasoned (paraphrasing the writeup): the `claude-agent-sdk` opens a long-lived session and emits a structured stream `SystemMessage(init) → AssistantMessage(text|tool) → ResultMessage(success|error)`. OneAPI relays in the wild typically expose only the `/v1/messages` REST surface, not the session orchestration plane. So the SDK would either timeout on init or be told "not logged in."

R71 measurements partially confirmed this: a default `query(prompt='hi', options=ClaudeAgentOptions(max_turns=1))` returned `AssistantMessage(error='authentication_failed', text='Not logged in · Please run /login')` against the relay. R71 took that as evidence the relay doesn't speak session protocol and gated alpha release on "real Anthropic key."

**R73's correction.** Re-running the same probe with one change — `ClaudeAgentOptions(max_turns=1, model="Claude Sonnet 4.6")` instead of leaving model unset — produces:

```
[T2] Probing query() stream (budget 30s)...
     message classes seen: ['SystemMessage', 'AssistantMessage', 'ResultMessage']
     block classes seen:   ['TextBlock']
     saw assistant:        True
     synthetic model:      False
     auth failed:          False
     first text:           'pong'
```

What R71 actually hit was the SDK's **client-side fallback path**: when no `model` is set, the SDK probes a default model id (`claude-sonnet-4-5` or similar kebab-case form). The OneAPI relay rejects that id (its catalog uses `"Claude Sonnet 4.6"` with spaces and capitalization), and the SDK surfaces the rejection as a synthetic `not_logged_in` AssistantMessage rather than an error stream. R71 read the synthetic auth-failure blob and concluded the relay was hostile; it was actually the model id that was wrong.

The session protocol itself round-trips fine. ADR-027 ("replay-seam over a non-session relay") that R69 sketched as a contingency is therefore **not needed** — the contingency it was guarding against does not occur. R74 will not write ADR-027.

### Why R69 missed this

R69 spike #1 ran the SDK against the relay without a model parameter (matching the SDK doc examples). It then attributed the resulting `not_logged_in` message to relay incompatibility, because that's what the message text says. The fix is mechanical and clear in retrospect, but in R69 the failure shape (auth) and the predicted failure shape (auth-or-protocol) were the same, so the wrong root cause stuck. **R73 added a model parameter on the dogfood probe; that single line of code reversed the round-69-through-72 conclusion.**

### Process lesson recorded

The R69 → R71 → R72 chain shows a class of error worth naming: an invariant that flows from one round's research into the next round's progress doc into the next round's release-gate language can survive multiple sanity checks before a fresh probe finds it false. The remedy is cheap and we should fold it into the release-pattern: **before any release that gates on a previous round's un-tested research conclusion, re-run the smallest possible disprover.** I'll patch `chronos-release-pattern` skill below.

## Release: v0.7.0a1 (alpha)

This is Chronos's first explicit pre-release. Per ADR-026 §"Release plan", Arc B ships incrementally:

- **slice 1 (this release)**: scaffold + recorder + live-smoke + dogfood probe. **Read-only**; no `fork_session()` integration yet.
- slice 2 (R74-R75): SDK-native `fork()` delegate (the SDK exposes `client.fork_session()`).
- slice 3 (R76+): tool-call dispatch, MCP server passthrough, multi-turn hardening.

**Why alpha**: the recorder works on the happy single-turn path; the multi-turn / tool-use / MCP paths have characterization tests but no live coverage. Anyone trying Arc B in a real workflow this week needs to know it's a peek, not a contract.

**Dist**: Python 3.11+ wheel, sdist. Frontend bundle unchanged from 0.6.0 (no UI surface for the new adapter yet — R76 lights that up).

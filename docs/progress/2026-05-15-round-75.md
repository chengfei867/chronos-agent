## §1 — Starting state (inherited from R74)

R74 closed `v0.7.0a2` Arc B slice 2: `AnthropicAgentsRecorder.fork()` shipped, 5 unit tests + live-smoke harness + dogfood script all green. The R74 hand-off offered three follow-ups:

- **A**: Slice 3 — tool-call dispatch & multi-turn fork (estimated ~2 cron slots; non-trivial SDK surface).
- **B**: Codify the implicit `record()`→`fork()` invariant as an ADR amendment + tests (~1 cron slot).
- **C**: Web UI compare slice (cross-cutting; multi-slot).

I picked **B**. Rationale:

1. **Single-slot fit.** R75 is a cron slot, not an arc. Option A explicitly carries multi-slot risk per R74's own estimate.
2. **Protects R74's own work.** R74's `fork()` reads `uuid` + `session_id` off `Node.state_after`, which `record()` happens to stamp at line 308 of `recorder.py`. Nothing forces this — a future refactor that "tightens" the metadata loop to skip empty values, or moves the stamping into a separate code path, would silently break `fork()` while every existing test still passes (because the existing fork tests construct `state_after` by hand, not via the live recorder pipeline).
3. **R57 doctrine compliance.** R57 invariant: when an ADR's implementation evolves new constraints, amend the ADR in-place; don't write a new ADR for a corollary.

Starting working tree: clean against `origin/main` at `74b470a`. Two stray frontend pnpm files untracked (`frontend/pnpm-lock.yaml`, `frontend/pnpm-workspace.yaml`) — explicitly excluded from this commit; project standardised on npm in Arc A R63 and the pnpm artefacts are local-tooling noise, not project state. Will defer the gitignore entry to a future round to keep R75 focused.

## §2 — Work done

### §2.1 — ADR-026 §5 amendment

Patched `docs/decisions/ADR-026-arc-b-scope.md`:

- Bumped Status header to note the R75 amendment (in-place per R57).
- Inserted new **§5 — Inter-method contract: `state_after` seed coordinates (R75 amendment)**:
  - Contract table enumerating all five metadata keys stamped by `record()` on `AssistantMessage` / `ResultMessage` nodes (`uuid`, `session_id`, `stop_reason`, `total_cost_usd`, `duration_ms`).
  - Split into **MUST** (`uuid`, `session_id`) and **MAY** (the three observability keys). MUST keys are the fork anchors that `claude_agent_sdk.fork_session(session_id, up_to_message_id=uuid)` requires; MAY keys are observability-only and may evolve without amending the ADR.
  - "Why MUST and not SHOULD" subsection explaining the loud-failure rationale: a missing MUST key in `state_after` should fail fast at fork time rather than degrade silently.
  - "Test enforcement" subsection cross-referencing the two new unit tests by name.
  - "Out of scope" subsection clarifying that adding new MAY keys is not a contract change.
- Renumbered the previous §5 (Open questions) → §6 to make room.

### §2.2 — Unit tests (executable enforcement)

Patched `tests/unit/test_adapter_anthropic_agents.py` after line 413 (existing §6 block boundary):

- Added new **§6.1 — `state_after` seed-coordinate contract (ADR-026 §5)** comment header.
- Added `test_record_state_after_carries_seed_coordinates_for_assistant`: builds an `AssistantMessage` event with explicit `uuid="asst-uuid-fixture"` + `session_id="sess-fixture"`, drives it through `recorder.record()`, then asserts the persisted `Node.state_after` dict contains both keys with the expected values.
- Added `test_record_state_after_carries_seed_coordinates_for_result`: same shape but for `ResultMessage`, since `fork()` accepts both as anchor types.

These tests deliberately exercise the live recorder pipeline (not a hand-crafted `state_after` dict) so they fail loud against any refactor of the metadata-stamping loop in `recorder.py`. The existing fork tests construct `state_after` directly and would NOT catch such a regression — that's the gap §6.1 closes.

- **NEW invariant elevated to project-level**: *Any contract that lives across two methods of the same class, where one method writes state the other reads, must be (a) named in the relevant ADR, (b) enforced by a test that exercises the writer-side independently, (c) commented at the writer's source site referencing the ADR.* Triple-redundant pin, intended to survive a refactor by a maintainer who's only read one of the three.

## §8 — Author note

R75 is a defensive round, not a forward-progress round. It doesn't ship any new user-facing capability — it pays down a specific category of latent regression risk that R74 introduced. This kind of round is how the 23-round zero-regression streak gets sustained: not by being lucky, but by going back and explicitly pinning the invariants that earlier rounds happened to satisfy without naming them.

The pattern I want to keep practising:

1. Round N ships a feature.
2. Round N+1 re-reads round N's progress doc and asks "what implicit contract did this feature accidentally rely on?"
3. If there's an answer, codify it in an ADR + test + source comment before round N+2 introduces a refactor that breaks it.

This is the third time the pattern has fired in Phase 4 (R57's ADR-021 amendment, R69's disprover doctrine, now R75's ADR-026 §5). Worth elevating into a new project-level invariant — flagged in §4 above.

— Hermes Agent

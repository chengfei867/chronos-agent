**§5.3 amendment** inserted before `### 6.` (line 461). Major sub-sections:

- **Contract** — `tool_result_overrides: dict[str, Any] | None = None` kwarg on `fork()`. Key = `tool_use_id` (same JOIN anchor as §5.1 / §5.1.1). Value = JSON-serialisable Python (typically `str` or list of content blocks). None / `{}` are no-ops.
- **Validation** (mirrors §5.2; runs before SDK call):
    1. **Key-type**: must be `str` → `AdapterError`.
    2. **Result-side keyset membership**: key must appear in the union of tool-use ids on `_is_result_side` Nodes (UserMessage carrying ToolResultBlock per §5.1 / element of plural §5.1.1). An id present *only* on use-side (orphan use, no matching result yet) is rejected here as "not in result-side keyset" — slice-3a→3c coupling pre-condition (round-trip must close in parent run).
    3. **No double-substitution**: a key MUST NOT also appear in `tool_input_overrides`. Rewriting both input and result for the same `tool_use_id` is contradictory (input rewrite implies re-invocation; result rewrite implies non-invocation). `AdapterError` quoting the conflicting id.
- Documented the **asymmetry vs §5.2**: §5.2 validates against use-side keyset and rejects orphan use-ids; §5.3 validates against result-side keyset and rejects use-only ids. A round-trip that closed in the parent passes both.
- **Child-side stamp**: `state_after['tool_result_content']` (singular) for §5.1 result Nodes; `state_after['tool_result_contents']` (plural, index-aligned) for §5.1.1 multi-block result Nodes. Mutually exclusive, same `len==1 / len>1` rule. JOIN anchors `tool_use_id` / `tool_use_ids` preserved byte-for-byte.
- **Out of scope** for slice 3c: HTTP/CLI surface (R82 is Python-API-only); multi-step result substitutions across nested forks; MCP-aware filtering; streaming / partial substitution within a multi-block result.
- **SQL recipe**: `SELECT ... WHERE json_extract(state_after, '$.tool_result_content') IS NOT NULL`. Multi-block consumers swap to `json_each` over `tool_result_contents`.
- **Test enforcement** section enumerates the 4-test plan with strict-xfail markers and links the R81 scaffold file.

## Decisions & rationale

### D1 — Pick spec+scaffold over consolidation or slice 4

R80 progress doc lines 162-167 nominated three options. Picked **Option 1 (§5.3 amendment + scaffold)**:

- Preserves the slice-by-slice cadence (R52→R81 = 29 rounds zero-regression).
- Keeps the strict-xfail forcing function active across rounds (R76→R77 §5.1.1, R79→R80 §5.2, now R81→R82 §5.3).
- Consolidation pass (Option 2 — extract `_StubBlock` / `_StubMessage` fixtures, now FOUR duplications) should wait until slice 3 fully closed end-to-end. Doing it mid-slice risks merge conflicts with the R82 implementation diff.
- Slice 4 (Option 3) is premature — slice 3 hasn't fully shipped (3c still open).

### D2 — Validation #2 anchors against result-side keyset, not use-side

**This is a deliberate deviation from the CONTEXT.md §6 R81 plan.** That plan (lines 722, 725) sketched §5.3 as allowing orphan-use-id substitution — "the user might want to inject a result for a tool_use that didn't get one in the parent" — and listed Test 4 as `test_fork_with_result_override_of_orphan_use_id_succeeds` (strict-xfail, expected to *succeed* under R82).

I shipped a stricter contract instead. Rationale:

- The natural intuition was "use the same use-side keyset as §5.2 since they're keyed by the same `tool_use_id`". I rejected that because the user story for §5.3 is "the tool ALREADY closed the round-trip in the parent — replay reasoning under a hypothetical result". An orphan use-id (R76's tolerance clause) means the round-trip *didn't* close in the parent → there's no `ToolResultBlock` Node in the parent run for the override to "rewrite" → the override doesn't have a clear binding target.
- "Inject a result that didn't exist in the parent" is a different semantic primitive than "rewrite an existing result". The former is closer to a test-double / mock injection; the latter is replay-with-substitution. Conflating them in one kwarg invites confusion downstream when consumers SQL-enumerate substituted calls.
- A future slice 3d (if real demand surfaces) can introduce a separate `tool_result_injections` kwarg with explicit "no parent result Node" semantics. Keeping §5.3 strictly mirror-symmetric to §5.2 (both reject half-open round-trips) makes ADR-026 §5 a coherent block.

The slice-3a→3c coupling pre-condition is therefore the same as slice-3a→3b's: round-trip must close in parent run before fork-with-substitution can rewrite either side. Both §5.2 and §5.3 reject half-open round-trips, just on different sides.

**Test 4 also changed shape**: where the CONTEXT plan had a 4th xfail "orphan-id succeeds" test, I shipped a 4th xfail `test_fork_with_result_override_collides_with_input_override_raises` instead — a §5.3-unique validation #3 (no double-substitution) that the CONTEXT plan didn't anticipate. The orphan-id-succeeds case is **deferred indefinitely** pending real user demand; if it shows up, R83+ ships it as `tool_result_injections` (separate kwarg) per the rationale above.

This deviation is logged here per project convention (direction drift must be argued in progress doc).

### D3 — Validation #3 (no double-substitution) is §5.3-unique

§5.2 has no equivalent of "no double-substitution" — its validation list is just key-type / unknown-id / orphan-id. §5.3 adds the new collision check because *both* override mappings can land on the same `fork()` call and the semantics contradict (input rewrite ⇒ re-invocation; result rewrite ⇒ non-invocation). Catching this fail-fast in R82 is cheaper than letting downstream consumers reason about a self-contradictory state.

### D4 — Defer fixture extraction to R82 / follow-up

This is the FOURTH file replicating `_StubBlock` / `_StubMessage` / `_aiter` (after `test_adapter_anthropic_agents.py`, `test_queries_tool_linkage.py`, `test_anthropic_agents_fork_tool_override.py`). The R58 / R78 convention (extract at 3 duplications) is now exceeded. Deliberately deferred:

- TDD scaffold rounds should not also do cross-file refactors (scope discipline).
- The R82 implementation diff will already touch `recorder.py:fork()` and the new test file; bundling a 4-file refactor invites merge friction.

Tracked as next-round defensive TODO. R82 (post-impl) or a dedicated cleanup round should extract to `tests/unit/fixtures/anthropic_agents.py`.

### D5 — Stamp keys named `tool_result_content` / `tool_result_contents`

§5.2 used `tool_input` / `tool_inputs`. The natural symmetric pair would be `tool_result` / `tool_results` — but I chose `tool_result_content` / `tool_result_contents` to:

- Mirror the SDK's `ToolResultBlock.content` field name exactly (consumer recipes stay grep-able from SDK docs to chronos state_after).
- Avoid collision with any future `tool_result` aggregate / metadata field.
- Surface the fact that the override replaces the **content** payload specifically, not the entire ToolResultBlock structure (which retains its `tool_use_id` JOIN anchor on the SDK-side block; only `content` is overridden).

Documented in §5.3 #### Child-side stamp.

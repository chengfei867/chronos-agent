### 2. Probe: does the OneAPI relay support MCP tool-use?

R83's deferral note hypothesized AC-2 needed:

> *"…an MCP server fixture + Node.js subprocess on the runner."*

I ran a 30-line probe to test the assumption (one `create_sdk_mcp_server` + `tool` decorator + `query()` against the live `Claude Sonnet 4.6` model):

```
SystemMessage 'init' …
AssistantMessage ThinkingBlock …
AssistantMessage ToolUseBlock { name=mcp__math__add, input={a:4127,b:8956}, id=toolu_bdrk_… }
UserMessage     ToolResultBlock { tool_use_id=toolu_bdrk_…, content=[{type:text,text:13083}] }
AssistantMessage TextBlock "The result of **4127 + 8956 = 13,083**."
ResultMessage 'success'
```

**Three findings collapsed three deferred risks into a one-round fix:**

1. **The OneAPI relay surfaces `ToolUseBlock` and `ToolResultBlock` to the SDK end-to-end.** Earlier doubt was that the relay might only support flat text (per R64 doc that called the path "session protocol only"). It doesn't — full tool-use protocol works.
2. **`claude_agent_sdk.create_sdk_mcp_server` makes Node.js entirely unnecessary.** Tools are ordinary Python `async def` functions decorated with `@tool(...)`; the server runs in-process, no subprocess.
3. **The recorder already handles ToolUseBlock + ToolResultBlock correctly** (R76 wiring): `tool_use_id` lands in `state_after`, IDs match across blocks.

### 3. Wrote the AC-2 dogfood

`scripts/dogfood/arc_b_slice_3_mcp.py` (~280 LOC):

- Sets up a 1-tool MCP server: `add(a: int, b: int) -> int`. Operands are non-trivial (4127 + 8956 = 13083) so the model must actually use the tool, not memorize.
- Builds `ClaudeAgentOptions(max_turns=4, model='Claude Sonnet 4.6', mcp_servers={'math': server}, allowed_tools=['mcp__math__add'], system_prompt=…)`.
- Drives `query()` through `AnthropicAgentsRecorder` with the **R71 sync-recorder pattern** (build async iterable in this loop, hand recorder to `run_in_executor` to dodge `asyncio.run() cannot be called from a running event loop`).
- Inspects recorded nodes; computes a structured report; runs five invariant checks; exits 0 only if all pass.
- Outer 120s timeout guard + the same wallclock guard around the recorder coroutine.

**First run failed.** The dogfood's INV-2/3/3b/4 all tripped. Debugger printed `state_after` for every node — discovered the recorder writes block dicts with key `"block"` (not `"type"`), and stamps `kind=NodeKind.LLM` (not `TOOL`) on Assistant/User messages even when they carry `ToolUseBlock`/`ToolResultBlock`. **The recorder's `kind` comes from message-type dispatch (recorder.py:27-46), not block-type dispatch.** The `NodeKind.TOOL` entry in the block-dispatch table at recorder.py:77 is unused for these blocks.

This is **not a bug for AC-2**: `tool_use_id` IS in `state_after`, the R76 linkage contract holds, the data is fully recoverable. But it IS a contract finding worth documenting. I:

- Updated the dogfood's invariant inspector to filter on `state_after['blocks'][i]['block'] in {'ToolUseBlock','ToolResultBlock'}` rather than `n.kind == NodeKind.TOOL`.
- Tolerated thousands-separator rendering ("13,083" vs "13083") since the model wrote the latter.
- Documented the contract finding inline in the inspector docstring.
- Deferred reconciliation/contract-doc to a future round (CONTEXT §6 R86 TODO).

Second run: **`AC-2 release-gate INVARIANTS GREEN`**, exit 0, run took ~7 seconds, cost reported as $0.14.

## Decisions

### D1 — Picked Option A (AC-2 GA-gate close) over Option B (visible-range pagination polish) and Option C (recorder contract docs)

CONTEXT §6 listed all three. Option A had the highest leverage despite being assumed-hard (per R83's deferral note) because:

1. AC-2 is one of two GA-blocking checkboxes — every round it stays open is a round v0.7.0 GA stays gated.
2. R85 is a 3-hour cron slot, ample for a focused GA-gate close if the path is clear.
3. The deferral note's hypothesis (Node.js subprocess) was untested. A 30-line probe is cheap; if the assumption was wrong, AC-2 was a one-round task.

The probe took ~3 minutes. Confirmed Node.js wasn't needed. Locked in Option A.

### D2 — Used `claude_agent_sdk.create_sdk_mcp_server` (in-process) instead of an external MCP server

Three reasons:

1. **Zero subprocess dependency.** No `npx`, no `npm install`, no Node version pinning, no PATH check, no fork-bomb worry on cron retry. Ports CI runners trivially.
2. **Test isolation.** Tool runs in the same Python process as the recorder; no race between SDK launching subprocess and the dogfood asserting on captured shape.
3. **Sufficient for AC-2.** The acceptance criterion is "≥1 MCP tool". One in-process tool ticks the box. Fancier multi-tool / external-server scenarios are post-GA polish.

Recorded in dogfood module docstring; the alternative (Node MCP server) is the natural future direction if/when we need to certify against external MCP servers.

### D3 — Did NOT fix the recorder contract finding (`kind=LLM` for ToolUseBlock messages)

Found a discrepancy: recorder's block-dispatch table at recorder.py:77 has `"ToolUseBlock": NodeKind.TOOL`, but the actual `kind` stamped on the recorded node is `NodeKind.LLM` because kind comes from message-type dispatch, not block-type.

**Why deferred:**
- Not blocking AC-2: `tool_use_id` is correctly in `state_after`, R76 linkage works, the dogfood's five invariants pass with the observed shape.
- Touching this is a recorder behavioral change. R85 budget is for AC-2 close, not adapter refactor. Mixing them risks regressing the 33-round streak.
- Reconciliation has two valid resolutions: (a) document `kind=LLM` as intentional in the AdapterProtocol contract (because messages contain blocks, kind reflects the message envelope); (b) split nodes so each block becomes its own node with the correct kind. Choosing between (a) and (b) deserves a focused round + ADR.

Recorded in dogfood docstring + ADR-026 §6 AC-2 closing note. CONTEXT §6 carries it forward as an R86 candidate (low-priority since AC-2 is closed).

### D4 — Tolerated `13,083` and `13083` in the final TextBlock invariant

Live model rendered the sum with thousands separator. INV-4 must accept both forms or it'd be flaky on a benign rendering choice. Encoded in the invariant: check both `str(_EXPECTED_SUM)` and `f"{_EXPECTED_SUM:,}"`.

## Findings (R88 invariants for the wall)

1. **Aspirational-release-doc trap has TWO variants — codify both** (this round). R86 = "true aspiration" (gate evidence missing). R88 = "partial execution" (gate evidence intact; only release-engineering steps missing). Same trap shape pre-flight, different recovery prescription. Distinguished by the gate-evidence-valid axis. Skill now documents both with a 7-row diagnostic table. Pre-flight check at every round start: `git status` (any un-pushed commits?) + `git tag --list "vX.Y.Z*"` (any expected tags missing?) + `releases/latest` API (does it match the latest CHANGELOG block?). The 60-second check that caught R88 belongs in every cron round's setup phase, not just release rounds.

2. **The 8-step `chronos-release-pattern` is not atomic — last 2 steps (push, Release POST) can silently no-op**. R87's progress doc shows P9 + P10 written as if they ran, but git history + remote state proved otherwise. Possible causes: (a) cron-slot timeout near round end, (b) network blip on the gh-proxy or api.github.com calls, (c) progress-doc was written speculatively before P9/P10 ran. No way to fully diagnose post-mortem since R87 didn't preserve shell logs. Mitigation: post-round verification step in `chronos-release-pattern` skill — "after committing, fetch from remote and re-verify (a) commit on origin, (b) tag on remote, (c) GET releases/latest matches new tag". Will patch the skill in R88 close-out if not already there.

3. **Even pristine progress docs can mis-report execution state** — R87's progress doc is one of the cleanest in the project (87 lines, well-structured, every section filled). And it still claimed completion of two steps that didn't run on the remote. **Lesson: progress docs are intent + claim, not ground truth. Ground truth = git history + remote API state.** Going forward, if a release round's progress doc says "tag pushed + Release page created", the next round's setup MUST re-verify both via remote queries (cheap: one `git ls-remote --tags` + one `releases/latest` API call, ~1s each).

4. **Cost of recovery vs. cost of trap**: total R88 cost = 0 LLM (no live-smoke), 0 gate-flake risk (re-ran gates locally, all green), ~5 minutes wallclock. Total R88 value = v0.7.0 visible as "Latest" on GitHub for the first time, install instructions reach reality (`pip install chronos-agent==0.7.0` would actually resolve once published — repo still private but tag is now real), skill hardened. **Recovery rounds are cheap; un-recovered traps are expensive (every future round's pre-flight burns recognition cost re-discovering them, plus user-trust erosion if the user inspects GitHub and sees v0.6.0 still latest)**. Always-recover-immediately is the right default for this class of trap.

### 3. ADR-026 §5.2 amendment

Inserted a new section between §5.1.1 (line 291) and "### 6. What
Arc B slice 2+ looks like" (now line 461). Sibling-extends, does NOT
supersede — per R57 in-place ADR amendment invariant.

§5.2 specifies:

- **Contract**: `fork(..., tool_input_overrides: dict[str, dict[str,
  Any]] | None = None)`. Mapping `tool_use_id → new_input_dict`.
  Both singular (R76 §5.1) and plural (R77 §5.1.1) anchors are valid
  keys (per-id granular, not per-Node).
- **Identity defaults**: `None` and `{}` are both identity forks —
  semantically equivalent to R74. Keeps R74 callers source-stable.
- **Stamp on child Nodes**:
  - `state_after['tool_use_id']` / element of
    `state_after['tool_use_ids']` — UNCHANGED. JOIN anchor binding
    contract preserved across fork.
  - `state_after['tool_input']` — NEW. Substituted dict; absent on
    Nodes that did not have their input rewritten. For multi-block
    nodes, this is a list aligned by index with `tool_use_ids`,
    `null` entries for verbatim blocks.
- **Validation (fail-fast)**: Three rules — key-type, unknown-id,
  orphan-use-id (using `chronos.queries.tool_linkage.unmatched_tool_uses`
  for the slice-3a→3b coupling). All raise `AdapterError` *before*
  the SDK fork_session call.
- **Test enforcement**: References the four R79 tests by name +
  rationale.
- **Consumer SQL pattern**: Singular form (`json_extract(state_after,
  '$.tool_input')`) and plural form (`json_each` with index-aligned
  `je.key = ji.key` per §5.1.1).
- **Out of scope**: ToolResultBlock substitution (slice 3c / §5.3),
  multi-step substitutions in one call (supported but discouraged),
  HTTP/CLI surface (separate slice), `chronos.queries.tool_linkage`
  promotion (still internal until dedicated ADR).

This is the load-bearing slice-3a → slice-3b coupling: R78's
`unmatched_tool_uses` helper exists *for this validation rule*. Slice
3a's reader-side closure (R78) is now actively used by the §5.2 spec —
exactly the two-layer architecture (frozen contract = SQL recipe in
ADR / mutable convenience = Python helper) ratified at R78.

R80 should pick whichever is on the critical path.

### D2. Strict-xfail vs. skip

Per the plan, I went with `pytest.mark.xfail(strict=True, reason="slice
3b — R80")` over `pytest.mark.skip`. Rationale (was: "Recommend
xfail(strict=True) — 主动守 R80 实施 (xfail-but-pass = R80 done & 提示
删掉标记)"):

- `skip` is silent — R80 agent can land the implementation and forget
  to delete the markers; tests stay skipped, no signal.
- `xfail(strict=True)` is loud — when impl flips them to passing,
  the strict mode trips and CI goes red. The R80 agent **must**
  delete the markers to make CI green again. Built-in completeness
  check.

### D3. Identity guard test on R79 (not xfail)

The plan called out test #1 (`test_fork_without_overrides_is_identity`)
as "sanity, expected pass". I initially marked it xfail like the others
and the run flagged `XPASS(strict)` → strict-xfail trip.

Reasoning: my R79 pass-through accepts the kwarg and falls through to
the R74 verbatim fork path when the mapping is None / empty. So the
identity contract is **already satisfied** in R79; the test should
PASS (not xfail) and act as a regression guard during R80 development
(if R80 accidentally breaks identity by always running the substitution
pipeline, this test trips first).

Removed the xfail marker on test #1, kept it on tests #2-4. Result:
**1 pass + 3 xfailed**, exactly as ADR §5.2 ####Test enforcement
prescribes.

### D4. Validation #3 wording — "use-side orphan" not "any orphan"

The ADR specifies validation #3 as "appears in `unmatched_tool_uses`",
which is the **use-side** orphan helper. There's a symmetric
`unmatched_tool_results` (result-side orphan), but slice 3b only
needs the use-side check because:

- Slice 3b rewrites `ToolUseBlock.input` — the asymmetric concern is
  "is the use side currently pending a result?" not "is some random
  result orphaned?".
- A result-side orphan (use missing for a recorded result) doesn't
  block slice 3b — those are typically "recorder picked up a resumed
  session mid-conversation" cases (§5.1's fourth contract clause),
  unrelated to whether you can rewind the use side.

Test #4 is named `test_fork_with_override_of_orphan_use_id_raises` to
make this precise (not `..._of_orphan_id_raises`).

## Findings

### F1. Context compaction can misalign on round identity

When mid-round context is compacted, the summary may misremember the
**actual** task plan if the compactor skews toward recently-loaded
files. In R79's case, the compaction summary said "slice 3b query
layer / `tool_pairs` module / `test_queries_tool_linkage.py`" — but
CONTEXT.md §6 is the source of truth and clearly says "slice 3b TDD
scaffolding for fork-with-tool-substitution / `test_anthropic_agents_
fork_tool_override.py`".

**Recovery recipe** (worked here):

1. After context compaction, the **first** action MUST be re-reading
   `docs/CONTEXT.md` §5 + §6 fresh. Do not trust the summary's task
   description.
2. Cross-check by `git log --oneline -10` to confirm the actual HEAD
   (the summary may also be wrong about what shipped).
3. If the summary disagreed with CONTEXT, log the discrepancy in the
   progress doc (this section) so future rounds know the compaction
   layer can drift.

This is round-handoff hygiene that scales: even if the human-written
CONTEXT and the agent-written round summary agree, a context
compaction layer between them can introduce drift.

### F2. ruff `SIM117` nested-with auto-fix is safe

In tests #3 and #4, I wrote:

```python
with pytest.raises(AdapterError, match="..."):
    with recorder.fork(...):
        pytest.fail("...")
```

ruff flagged `SIM117` (combine with statements). `--fix` collapsed
them into a single `with pytest.raises(...), recorder.fork(...):`
block, which is semantically equivalent and more idiomatic. No
behavior change, all tests still red on the same line. **Auto-fix is
safe for `SIM117` in pytest.raises scaffolds**.

### F3. Slice-3a → slice-3b coupling validates two-layer architecture

R78 shipped `unmatched_tool_uses` as an **internal** helper, with the
two-layer architecture pinned: SQL recipe in ADR is contractual
surface, Python helper is mutable convenience. R79 is the first round
to *use* the helper — and it does so as part of an ADR amendment
(§5.2 validation #3). The internal helper is now load-bearing for a
later contract.

This validates the R78 F1 prediction: "Helper is internal — may
evolve freely between minor versions; SQL recipe is the contractual
surface. Reusable when a future ADR amendment adds a new JSON-bag key
(slice 3b/3c will follow this pattern)." R79 §5.2 is the first
follow-on. Slice 3c (§5.3, MCP passthrough) likely the second.

The clarification: when an ADR pins behavior that *uses* the internal
helper (rather than just defining a new bag key), the helper's
identity-of-callers becomes part of the ADR's contract. R79 §5.2's
validation #3 now explicitly mentions `chronos.queries.tool_linkage._
ids_from_state_after` and `unmatched_tool_uses` by name — so even
though they're "internal", their semantics are pinned by the ADR.
**Internal-API mutability has soft limits**.

### F4. Strict-xfail as a forcing-function pattern

Saving this for re-use across the project: when shipping a TDD
scaffold round (spec + red tests, impl deferred to next round), use
`pytest.mark.xfail(strict=True, reason="slice X — R{N+1}")` rather
than `pytest.mark.skip`. The strict mode means any accidental "skip
removal" by the impl agent without removing the marker → CI red.
This is a structural completeness check that doesn't rely on the
impl agent reading the round-handoff invariants section.

### F5. The "fork-and-submit_runtime" test ergonomics

`recorder.fork()` is a context manager that yields a `ForkRef`, and
the user calls `ref.submit_runtime(child_runtime)` *inside* the
`with` block. This is fine for happy-path tests but awkward for
"fork should raise before yielding" tests — the `with` block must
not be entered at all if validation fails.

Solution shape (used in tests #3 and #4):

```python
with pytest.raises(AdapterError, match="..."), recorder.fork(...):
    pytest.fail("fork() should have raised AdapterError before yielding")
```

`pytest.raises` is the outer context, so a raise from `recorder.fork()`'s
`__enter__` (i.e. before the `yield`) propagates correctly and
`pytest.fail()` is unreachable. Confirmed in the xfail-strict path:
when R80 lands the validation, the raise will fire from the
`__enter__` site (after `_validate_overrides` runs but before the
`yield`), and pytest.raises catches it.

This is a useful pattern for any "validate-before-yield" context
manager test in the codebase. Worth promoting to the conftest /
fixtures as a helper.

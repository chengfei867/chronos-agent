## 2. Decision: why Arc A (recorded here for next-round readers)

The argument for Arc A over B/C is in ADR-023 §"Decision (R57 — Arc A
committed)" and won't be repeated in full here. Short version:

1. **Leverage** — Arc A turns already-recorded data (every run, every
   node, every fork across LangGraph/AutoGen/CrewAI) into *more
   insight* without adding adapter surface. Three adapters is enough
   breadth for 2026.
2. **Low risk** — ADR-006 two-run alignment is battle-tested since R39-A.
   N-run is a *pure function over N−1 copies* of that alignment.
   No new semantic model.
3. **Real internal demand** — during spike12/spike13 dogfooding I
   literally had three terminal tabs open running `chronos diff A B`,
   `diff A C`, `diff A D`. That's the user story.
4. **Arc B premature** — fourth adapter is breadth; we just bought
   breadth at v0.4.0. Diminishing returns unless a specific cohort asks.
5. **Arc C demand-driven** — zero external users, zero signal, so Docker
   / LAN-sharing / plugin system stay deferred.

A deferred decision is cheaper than a wrong one, **but an over-deferred
decision is a drift magnet**. R57 is the right slot: headroom exists,
signal is clearer than R56.

## 5. Key learnings to carry forward

### 5.1 `In-place ADR promotion is a valid pattern` (R57 new)

When a Draft ADR's Context + candidate framing remains correct and only
the decision changes, **promote in place** (Status + Decision + Follow-ups
rewritten; Context + Options preserved). Do NOT open a new ADR-024 that
duplicates the framing. The top-of-file Status line records the
history ("Accepted (skeleton R56; Arc A commit R57)"), so provenance
is preserved without double-maintenance. This applies to:

- Any "skeleton + decision-deferred" ADR pattern (see ADR-023 shape).
- Any ADR where the Options section is the reusable artifact.

When this *doesn't* apply: if the new decision invalidates the
original framing (e.g. rejects all listed options and invents a new
one), open a new ADR and mark the old Superseded.

### 5.2 `Phase-kickoff round archetype` (R57 new)

The R56 → R57 sequence reveals a two-round pattern for phase
transitions:

1. **Post-release polish round** (R56): close out previous phase on
   roadmap + README/guide adapter matrix refresh + draft-skeleton ADR
   for next phase with explicit decision-deferred.
2. **Phase-kickoff round** (R57): promote the skeleton ADR (or
   replace it), land the first slice's design doc, update roadmap
   priorities from "candidate" to "ACTIVE"/"DEFERRED". Zero code.

Both rounds are **docs-only**. Both are **single-slot**. Both
preserve the adapter zero-change streak.

Candidate skill: `chronos-phase-transition-pattern` — defer
authoring until the next phase transition gives a second data point
(currently only R56→R57 validates it; need R61→R62 or similar to
confirm the archetype).

### 5.3 `Design-doc Non-Goals section saves rounds` (R57 new)

`docs/design/n-run-compare.md` §2.2 has an explicit Non-Goals list
with seven ❌ entries. Writing this took <10 minutes and will prevent
at least one scope-creep conversation in R58/R59 (semantic diff is the
#1 risk — it's genuinely useful, but it has a *separate* ADR path).
Standard practice from here: any design doc ≥ 150 lines should have
a Non-Goals section.

### 5.4 `CLI-first → API-first → Web-optional` ordering for new features (R57 new)

The design doc explicitly orders delivery: CLI (R59) before API-via-CLI-
shape-lock (R59 same round) before Web (R60, optional). Rationale:
CLI *is* the JSON contract test. Writing the CLI first forces the
JSON shape to be decided before React props harden assumptions. This
inverts the instinct to ship Web first for demo-ability.

Applicable to: any new verb/endpoint pair. Applicable to: the next Arc A
second slice (fork-tree DAG viz) — same ordering planned.

## 7. Next round (R58) — binding per ADR-023 Follow-ups

**Primary**: implement `chronos.core.diff.merge_pivot_reports()`.

Specifically:

- New function signature (proposed, see design doc §9 for binding):
  ```python
  def merge_pivot_reports(
      pivot_run_id: str,
      other_run_ids: list[str],
      reports: list[DiffReport],  # length = len(other_run_ids)
  ) -> MergedPivotAlignment: ...
  ```
  where `MergedPivotAlignment` is a new dataclass in `diff.py` with
  `alignment[]`, `summary{}`, `warnings[]`.
- All inputs are already produced by existing code — this function
  is **pure**, no store access, no IO, no new deps.
- Tests (≥ 10):
  - N=2 reduces to identical `DiffReport.to_dict()` top-level summary counts.
  - N=3 with all equals → all `tag=equal` per cell.
  - N=3 with one changed in run B, another changed in run C at a
    different step → alignment row per step carries per-column tags.
  - Adapter-mismatch → `warnings[]` populated.
  - Duplicate ids in `other_run_ids` → raise (preserve §7.1 design
    decision).
  - `restrict_to_downstream` applied per-pair (one forked child + one
    unrelated run).
- Landing bar: `src/chronos/core/diff.py` gains ~150 LOC, `tests/unit/test_diff.py`
  gains ~200 LOC. No other files touched. Coverage floor: 94%.

### §2.2 — Option A P0: recorder linkage stamp

Patched `src/chronos/adapters/anthropic_agents/recorder.py` `_translate()`. After the existing `tool_blocks = [b for b in content if type(b).__name__ == "ToolUseBlock"]` collection (and the symmetric `tool_result_blocks` collection on the user-message branch), added two parallel stamps:

- AssistantMessage branch: when `len(tool_blocks) == 1` and `blk.id` is a non-empty string, stamp `state["tool_use_id"] = blk.id`.
- UserMessage branch: when `len(tool_result_blocks) == 1` and `blk.tool_use_id` is a non-empty string, stamp `state["tool_use_id"] = blk.tool_use_id`.

Both guarded by `isinstance(..., str) and value` to defend against SDK drift (None / empty / non-str). Comments cite ADR-026 §5.1 + R76 + the three enforcing tests by name.

Why symmetric on both sides: enables SQL self-JOIN across run nodes to reconstruct tool round-trip without parsing nested `state_after.blocks` list. The use Node and the matching result Node carry byte-identical `state_after['tool_use_id']`, so the JOIN becomes a single equality comparison.

Why not a new column: `state_after` is a JSON bag; queryable via `state_after->>'tool_use_id'` on SQLite. No migration, no schema review, no sidecar.

Why P0 only (single block): multi-block messages (>1 ToolUseBlock or >1 ToolResultBlock per message) are ambiguous for top-level linkage; deferred to a future P1 cut. SDK currently emits at most one tool block per message in observed traffic, so P0 covers 100% of real cases.

### §2.3 — Unit tests (executable enforcement)

Added `tests/unit/test_adapter_anthropic_agents.py` §6.2 with three tests:

1. `test_record_tool_use_block_persists_id` — AssistantMessage carrying single ToolUseBlock(id=`toolu_01ABCdefGhi`) → asst node's `state_after['tool_use_id']` equals that id; `tool_name` / `tool_input` columns also still populate (R70 contract preserved).
2. `test_record_tool_result_block_links_to_use` — full pair: assistant ToolUseBlock + user ToolResultBlock both carry the same `tool_use_id`; asserts byte-identical equality across the two Nodes (the JOIN anchor invariant).
3. `test_unmatched_tool_result_does_not_break_record` — orphan ToolResultBlock with no preceding AssistantMessage: `record()` MUST NOT fail (asymmetric tolerance — orphan is a valid resumed/forked-session entry condition); the result Node still surfaces `state_after['tool_use_id']`, allowing downstream JOIN to detect the orphan via empty match.

Side-tweak: `_StubBlockBase` dataclass got a new field `id: str | None = None`. Without this, `_blk("ToolUseBlock", id="...")` fails with TypeError before the test code can even exercise the recorder. Comment on the field cites ADR-026 §5.1 + R76 + slice 3a.

## §3 — Decisions / trade-offs

- **Combo round (D + A-P0) instead of single concern.** R75 was strictly single-concern. R76 takes a calculated relax: D is 4 lines and zero risk; A-P0 is the carve-out half of A. Combining them in one commit halves the per-change overhead without coupling concerns (revert-by-file is still trivial).
- **Slice 3a rather than full slice 3.** Full slice 3 wants tool-call dispatch + multi-turn fork + diff visualisation. None of those land in one cron slot. P0 lands the JOIN anchor that every slice-3 query needs as a cheap precondition; P1+ can build on it without re-doing recorder work.
- **No SDK install required.** The recorder still uses `type(b).__name__` duck-typing (R70 invariant). Tests use the existing `_StubBlockBase` dataclass — no SDK fixture, no live integration. Keeps unit suite hermetic per ADR-016.
- **JSON bag, not a schema column.** Considered adding `nodes.tool_use_id` column for index-friendly queries. Rejected: (a) `state_after` is the established carrier for cross-Node linkage (R75 precedent); (b) SQLite expression indexes on `json_extract(state_after, '$.tool_use_id')` give the same query speed without migration; (c) keeps slice 3a minimal.

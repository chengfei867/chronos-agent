### §1.1 ADR-024 Draft — Multi-pivot compare

`docs/decisions/ADR-024-multi-pivot-compare.md` (new, ~270 lines).

- **Status**: Draft (Arc A slice 4 framing).
- **Scope**: N-run compare when no designated pivot exists. Three realistic scenarios motivate it (variant sweep with no baseline, post-hoc comparison of independent sessions, ensemble/consensus view). v0.5.0's `merge_pivot_reports()` requires a pivot; if the user doesn't have one, they must either arbitrarily pick, run compare N times rotating pivot, or fall back to individual run views. Arc A slice 4 closes this gap.
- **Options considered**: A = pairwise O(N²) matrix, B = virtual pivot via MSA/LCS, C = auto-centroid star schema, D = hybrid A+C, E = defer.
- **Decision**: **Option C (auto-centroid) as slice 4 default**. Compute pairwise distances (counting metric, normalized to [0, 1]), pick `argmin` over runs of mean distance, call `merge_pivot_reports(pivot=centroid, others)`. Deterministic tie-break by lexicographic run_id. Option A deferred as **slice 5**, gated on slice 4 dogfood. Option B (MSA) explicitly rejected — ghost-run UX, empty-skeleton degenerate case, new-algorithm burden outweigh theoretical appeal. Option E (defer) rejected — R61 is a post-release planning slot with no competing urgency.
- **Distance metric** (`metric_version = 1`): `d(a, b) = disagreeing_positions / max(1, total_positions)` where positions come from ADR-006 diff verdicts. Cheap, bounded [0, 1], version-tagged for future refinement (ADR-025 candidate).
- **Consequences called out**: CLI flag surface decision deferred to R62 impl (likely `chronos compare --pivot=auto` with first positional arg optional); HTTP gets `?pivot=auto` query param; test matrix grows ~8-12 tests; distance-metric version becomes public contract needing its own discipline.
- **Naming-drift correction on the record**: CONTEXT.md §6 at R61 entry labeled this work "Arc B kickoff (Breadth: multi-pivot compare)", conflicting with ADR-023's `Arc B = Ecosystem (fourth adapter)`. ADR-024 reclaims the correct label: **this is Arc A continuation (slice 4), not a pivot to Arc B**. ADR-023's Arc B (Ecosystem) remains deferred and unchanged.

## §2 Why this shape (decisions on the record)

### §2.1 Why Option C over A, B, D, E

Full analysis in ADR-024 §Decision. Condensed rationale:

- **C vs. A**: matrix view is clean but `O(N²)` display doesn't scale past N≈10 and gives no ensemble-level single view; auto-centroid produces one principled merged view reusing v0.5.0 primitive verbatim.
- **C vs. B**: virtual pivot (MSA) produces ghost runs (UX alien), has catastrophic empty-skeleton failure on variant sweeps (common), and introduces heuristic alignment as a new algorithm project. Centroid sidesteps all three.
- **C vs. D (hybrid)**: D is "A + C shipped together". D's compute cost is same as A alone (distance matrix shared). The distinction is purely "ship both views now vs. ship centroid, gate matrix on evidence". R61 elects to ship evidence-gated matrix — if slice-4 dogfood shows cells the centroid hides, slice 5 ships; otherwise we save UI complexity we don't need.
- **C vs. E (defer)**: post-release planning is the highest-energy moment. Deferring loses momentum. Research doc + ADR is slot-sized work even without code.

### §2.2 Why naming-drift correction matters

CONTEXT.md §6 at R61 entry described this as "Arc B kickoff", but ADR-023's Arc taxonomy defines Arc B = Ecosystem (fourth adapter). ADR-024 explicitly rebinds this work as **Arc A slice 4** (continuation of Depth) to avoid two conflicting "Arc B" referents.

### §2.3 Why `metric_version` as a field

Distance metric (§3.3 in research doc) is a public contract: it affects which run is picked as centroid, which affects the rendered report. Refining the metric (kind-weighted? token-weighted? semantic?) should be an ADR-gated change. `metric_version` on every report makes the dependency explicit. Version = 1 from day one; ADR-025 candidate is "refine metric to v2" if/when dogfood shows v1 bias.

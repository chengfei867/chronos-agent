## 3. What was intentionally skipped / deferred

### 3.1 Screenshot refresh

CONTEXT §6 and the R48-B progress doc both flagged "refresh `docs/images/fork-modal/01-warning.png` (+ optionally 02 / 03) so the screenshots reflect the new badge icons" as an R48-C pre-cut checklist item.

**Deferred to R49.** Rationale:

- The R47-A screenshots were captured manually via a headful browser session. There is no automated Playwright / headless pipeline in `scripts/` for this view.
- A clean headless recapture inside a single cron slot costs ~15–25 minutes of wall time to set up: seed DB (`scripts/seed_r47a_effects.py`), boot the API (`chronos web`), spin up the Vite dev server, wait for hydration, drive three fork-modal opens, and capture three PNGs with the exact viewport/scroll the originals used.
- That time is all downside risk: if any stage hiccups (port collision, dev-server cold build, modal race), the release slips and we still need the screenshots anyway.
- Shipping the code bundle now keeps the tag discoverable, the CHANGELOG page honest, and the UX improvements installable via `uv pip install -e .` today. The screenshots are a documentation-site polish step, not a release blocker — the icons are visible to anyone who checks out the new tag and runs the UI.
- R49 can either automate the screenshot pipeline (worth it; three captures today, more when Phase 4 lands) or do a focused headful session, and in either case have more budget per round than the release-cut round.

The consequence is that for ~1 cron cycle, `docs/images/fork-modal/*.png` will show the pre-R48-B (single-color, no-icon) Alert. This is honest (the code that produced them was that code), and the prose in `forking-safely.md` is unchanged, so no text drifts.

### 3.2 i18n scan

CONTEXT §6 mentioned "可选: 轻扫 i18n — R47 说中文像机翻". A quick re-read of `frontend/src/i18n/zh.ts` `effects.*` and `forkModal.*` keys didn't surface anything obviously machine-sounding; the existing Chinese strings were hand-written in R44-A / R46-A and reviewed then. No change this round. If a future dogfood session surfaces specific phrasings, R49+ can patch them.

### 3.3 Additional tests

None. R48-A added 4 AutoGen classifier tests + a real-LLM spike. R48-B was pure UI with no schema change. The release itself needs no new tests; step-5 full-suite green run is the verification.

## 5. Decisions

### D-48-C-1: Defer screenshot refresh to R49 rather than block the release

Already litigated in §3.1. The short form: release cut is time-sensitive (want the tag out in the same cron window that edits the metadata), screenshot capture is not. Shipping `a2` now with prose/code accurate but hero image one-round-stale beats shipping `a2` tomorrow with both. Documented in CONTEXT §6 as R49's first item so it doesn't rot.

### D-48-C-2: Call the release alpha (`a2`), not beta

v0.4.0 non-alpha should wait until at least one real-world dogfood cycle against the full Phase 3 stack (AutoGen + LangGraph + effects + fork modal) runs cleanly. R49+ gives space for that without burning the `0.4.0` slot now. `prerelease=true` on the GitHub Release page.

### D-48-C-3: Theme string rewrite in the CLI status line

The `info()` line is user-facing prose, not a version number. Keeping "Phase 3 fork-safety bundle … forking-safely.md user guide" through v0.4.0a2 would undersell what actually shipped (two more rounds of work). Rewrote to "Phase 3 UX polish … effect-tag badge icons + AutoGen classifier fix (ADR-020)". Caught no stale assertions pinning the old string (verified via grep).

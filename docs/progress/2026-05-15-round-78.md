### §2.1 Decision: pure-Python over raw SQL

ADR-026 §5.1.1's SQL recipe is the canonical raw-query form (LEFT JOIN with `json_each`). My helper sits one layer above that, deliberately:

- Uses `store.get_nodes_for_run(run_id)` — a public SqliteStore method already in use across adapters.
- Builds two Python sets (use-side ids, result-side ids) in one pass.
- Returns `Node` rows in stream order whose declared id is absent from the complementary set.

Reasoning:
1. Keeps `SqliteStore` public API surface unchanged (no new method, no new column).
2. `_conn` is private — pure-Python path avoids the `# type: ignore`-prone privileged route.
3. ADR-026's SQL recipe remains the canonical raw form for consumers who *want* SQL (e.g., dashboards, CLI tools). The helper is the in-Python convenience for adapter-level / dogfood-script consumers who want a list-of-Node back without the JOIN ceremony.
4. O(N) load is fine for a debugging/observability helper — not on a hot path.

Trade-off accepted: this re-walks all Nodes per call (vs. an indexed SQL JOIN). Acceptable for the helper's stated audience.

### §2.2 Decision: new package `chronos.queries`

ADR-026 §5.1.1 calls these "consumer helpers"; CONTEXT §6 Option A spec'd `src/chronos/queries/`. First member of the package; reserved for future slices' read-side helpers. Followed R57 module-creation convention: empty `__init__.py` with re-exports + module file with full docstring.

### §2.3 No new ADR

CONTEXT §6 Option A explicitly said "不强求新 ADR; helper 是 internal API 而非 contract". Followed. Module docstring cites ADR-026 §5.1 / §5.1.1 verbatim and labels itself "internal — may evolve freely between minor versions".

## §6. Findings to wall

### F1 — Helper-vs-SQL split is the right shape for ADR-binding contracts

ADR-026 §5.1.1 pins SQL recipes as the canonical query form. R78 ships a Python helper *on top of* that, not *in place of* it. The helper is internal (`chronos.queries`, not exposed via HTTP/CLI/adapter contracts) and can evolve; the SQL recipe in ADR-026 is the contractual surface and stays frozen. Two-layer architecture: **frozen contract = SQL keys/shape; mutable convenience = Python helper that translates contract into idiomatic Python**. Reusable when a future ADR amendment adds a new JSON-bag key.

### F2 — `record()` pipeline as test fixture pays off again (R75/R76/R77/R78)

R75 instated "writer-side redundancy invariant: tests exercise live `record()`, not hand-crafted Nodes". R76, R77, and now R78 all honored it. Cost: each test file needs the stub message/block + `_aiter` helpers (~80 LOC duplicated). Benefit: any silent narrowing of `recorder.py:_translate()`'s metadata-stamp loop trips four downstream test files at once, not just `test_adapter_anthropic_agents.py`. **Pattern is now project-wide**: future read-side helpers also exercise `record()`.

If a fifth file ever needs the same stubs, extract to `tests/unit/conftest.py` or `tests/unit/fixtures/anthropic_agents.py` (R58 convention). Three duplications is the quorum I'd usually flag for extraction; we're at 2 (test_adapter_anthropic_agents.py + test_queries_tool_linkage.py) so still below threshold.

### F3 — Slice 3a's three-cut shape (P0/P1/P2 over R76/R77/R78) validates "read-side anchor first" sub-cut pattern

R76 finding F2 hypothesized the "read-side anchor first" pattern for sub-cutting big slices: ship the JOIN key (P0) before the multi-block extension (P1) before the consumer helpers (P2). R78 closes the third cut and the pattern holds — three single-slot rounds, each pure-additive, zero rework, three structural close-outs. **Confirmed**; should propagate to ADR-binding contract amendments going forward (slice 3b will be: P0 fork-API extension → P1 dogfood proof → P2 helper for "find substitution candidates").

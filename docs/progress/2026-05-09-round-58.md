## 2. What shipped

### 2.1 `src/chronos/core/diff.py` (+~240 LOC)

New public surface:

- `MergedCellTag` literal: `"equal" | "changed" | "removed" | "added" | "absent"`
- `@dataclass MergedPivotAlignment` with fields `pivot_run_id`, `other_run_ids`,
  `alignment`, `summary`, `warnings` + `to_dict()`.
- `merge_pivot_reports(pivot_run_id, other_run_ids, reports) -> MergedPivotAlignment`.

Key design decisions (binding for R59 CLI/API):

1. **No store access, no IO, no new deps** — pure function over the
   existing `DiffReport` dataclass. Caller (R59 wrappers) handles
   `store.get_run` / `diff_runs` / response framing.
2. **Row key is pivot `node.id`**, not `step_index`. This guards against
   the pathological case where two pivot entries share a step index;
   in practice step_index is unique per run but defensive keying costs
   nothing.
3. **Insert rows have `pivot_step=None` + `inserted_after_pivot_step: int`**
   (−1 sentinel = inserted before any pivot step). Rendering splices
   them after their anchor row.
4. **Cross-run insert merging**: if two different others insert a node
   with the same `node_name` at the same anchor step, they merge into
   one alignment row (two runs both added a "refine" step in the same
   position). Non-matching names stay separate rows. Heuristic but
   matches user intent for fork-sweep scenarios.
5. **"Absent" placeholders**: when a pivot row is mentioned by some
   reports but not others (can happen if a report is restricted to
   downstream past a fork point), missing runs get `{"tag": "absent"}`
   so every row has every other_run_id key populated.
6. **Input validation is strict**: empty `other_run_ids`, duplicates,
   pivot-in-others, length mismatch, report `run_a.id` mismatch,
   report `run_b.id` mismatch — all `ValueError`. Silent dedup / lax
   acceptance is a footgun (design doc §7.1).
7. **Adapter-mismatch warning**: populated per-pair in `warnings[]`,
   matches §4.3 of design doc. Non-fatal.

Complexity: N−1 linear passes over report entries, plus one sort on
pivot_step. O(N × L) work. No pairwise O(N²) diffs (design doc §4.1).

## 5. Findings (新上墙候选)

### 5.1 "Absent" cell placeholder is a real schema decision, not a filler

Initial draft had an insert-merge bug: when I pre-populated `per_run`
with `{"tag": "absent"}` for all non-contributing others, the merge
check `oid not in ir["per_run"]` always failed, so each other's insert
became its own row — producing 5 rows instead of the expected 4 on
the "both insert at same position" test. Fix: merge predicate must
accept a cell that is currently `absent` as "no contribution yet".
→ **"absent" is part of the merge algebra, not a placeholder.**

### 5.2 Fixture module at `tests/unit/fixtures/` is a right-sized abstraction

I considered putting the fixtures inline at the top of
`test_merge_pivot.py`. R59 will need the same fixtures for CLI and
API wrapper tests, and R61 dogfood will reuse them. Dedicated module
with `__init__.py` works. **Candidate pattern for cross-round test
reuse**: fixture modules under `tests/unit/fixtures/` (or
`tests/fixtures/` if integration tests need them too).

### 5.3 Strict input validation pays for itself immediately

Six of 17 tests are pure input-validation tests. Writing them caught
one real bug: my first draft didn't check `reports[i].run_b.id`
matching `other_run_ids[i]`. Without this check, a caller who passes
other_run_ids in wrong order relative to reports gets silently wrong
output (summary keyed on the wrong oid). Now `ValueError` immediately.
→ **For O(N) pure functions, over-validate inputs at the boundary.
The 6 tests are cheap; a silent miskey in a future CLI/API wrapper
is expensive.**

### 5.4 `RUF043` (raw strings for regex with `.`) bites pytest.raises

Ruff catches `match="run_a.id"` because `.` is a regex metacharacter.
Fix: `match=r"run_a\.id"`. Low-stakes aesthetic but still worth
knowing for future pytest-based validation tests. Add to lessons-
learned.

## 6. Decisions recorded (no new ADR needed)

No ADR churn — R58 is a pure implementation of ADR-023's Arc A first
slice as spec'd in `docs/design/n-run-compare.md` §4.1/§5.1/§7.1. The
design doc *is* the spec; this round binds it.

Internal structural decisions (below the ADR threshold):

- Insert-row merge heuristic: same anchor + same `node_name` ⇒ merge.
  This is *UX*, not semantics — the merge keeps the table compact for
  fork-sweep comparisons. Could later become configurable (`--no-merge-
  inserts` flag) if users report confusion. Defer until real feedback.
- "absent" as explicit cell tag vs. missing key in `per_run`: chose
  explicit. Every row has every other_run_id populated. Renderers
  (R59 CLI / R60 Web) don't need to handle missing keys.

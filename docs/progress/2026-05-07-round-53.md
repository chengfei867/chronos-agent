The probe found CrewAI 1.14.3 is a **source-compatible superset** of 0.80+ on every ADR-021 §D1–§D7 surface the recorder uses. The R52 scaffold needs zero code changes.

## 1. What triggered the pivot

### 1.1 Pre-flight discovery

The plan for this cron slot was CONTEXT §6 R53 P0 verbatim: write `tests/spikes/spike13_crewai_tool_effects.py` modeled on `spike10_autogen_tool_effects.py`, assert F1–F6, run against baidu-int OneAPI. Standard pre-flight:

```bash

$ set -a; source /workspace/.hermes/.env; set +a  # OneAPI key loaded
$ .venv/bin/python -c "import crewai; print(crewai.__version__)"
1.14.3
```

R52 pinned `crewai>=0.80,<1.0`. The pin can't be "respected" without `pip install 'crewai<1.0'` — and the installed 1.14.3 came from some Phase 2/3 transitive dependency (`uv sync` against the current lock). Dropping back to 0.95 meant fighting the resolver and locking Chronos to a past-EOL line. That was the fork in the road.

### 1.2 Two-minute triage per ADR-021 §D8's own language

Re-reading ADR-021 §D8:

> Pin upper bound at `< 1.0` pre-emptively — CrewAI hasn't cut a 1.0 yet and the event schema is explicitly marked unstable in their docs.

That's the decision, stated as pre-emptive, not empirical. A pre-emptive pin that's been falsified by reality is a **pin revision**, not a scaffold bug. This matches the precedent from ADR-013 / ADR-019 / ADR-020 — ADRs can be revised once empirical evidence replaces prior guesses.

### 1.3 The decision

- **R53 scope pivot**: instead of spike13 P0 (real LLM, 2-3 hr wall clock, first leg requires the pin fixed anyway), do ADR-022 + `spike13a_crewai14_event_bus_probe.py` + pin bump. Self-contained 30-min deliverable. R53 P0 slides to R54.
- **No change to ADR-021 D1–D7** — the pivot is a version-pin revision, not an architectural re-think. The D1–D7 event-bus design survives CrewAI 1.0 entirely.
- This is not drift. CONTEXT §6 explicitly lists "R53 known failure paths" and CrewAI version pin issues weren't among them — but CONTEXT §6 also says "If R53 discovers a real-world blocker, adjust scope and ratify with an ADR." That's exactly what we did.

## 2. spike13a findings (R52 scaffold × CrewAI 1.14.3)

`tests/spikes/spike13a_crewai14_event_bus_probe.py` is a **surface probe**, not a real-LLM spike. It imports the live `crewai` package, pokes every public API the R52 scaffold uses, fires one synthetic `ToolUsageStartedEvent` through the real bus, and drives `CrewAIRecorder.record()` end-to-end with a stubbed `_FakeCrew`.

| F# | Probe | Result on 1.14.3 |
|---|---|---|
| F0 | `crewai.__version__`, `crewai_event_bus` importable | 1.14.3, `CrewAIEventsBus` singleton present |
| F0 | `scoped_handlers`, `flush`, `on`, `emit` all on bus | ✅ all present with same signatures |
| F1 | `ToolUsage{Started,Finished}Event`, `LLMCall{Started,Completed}Event`, `Task{Started,Completed}Event` import from `crewai.events` | ✅ all import |
| F1 | `CrewKickoffCompletedEvent` from `crewai.events.types.crew_events` | ✅ imports (ADR-021 §D4 "optional" path is unnecessary on 1.14.3; the optional-import guard is still good to have for older lines) |
| F2 | `scoped_handlers()` CM attaches and detaches cleanly around a real `bus.emit(_Src(), event=ToolUsageStartedEvent(...))` | ✅ received=`['tus:probe_tool']`, bus flushed clean |
| F3 | R52 `crewai_adapter.build_recorder(store)` returns `CrewAIRecorder` and `record()` CM completes with a stubbed crew + 1 synthetic event | ✅ `run_id` + 1 node persisted |

Net: **zero surgery to the R52 scaffold needed**. The 32 duck tests in `test_adapter_crewai.py` continue to pass against the live 1.14.3 resolver because they use `_FakeEventBus` (which matches the real API shape verified above).

**Limitation acknowledged**: spike13a only tests the *surface*. It emits one synthetic event. It does not certify that a real `Crew.kickoff()` run produces well-formed `ToolUsage*Event` / `LLMCall*Event` payloads end-to-end with effect classification firing correctly. **That's R54 P0's job.** spike13a's only deliverable is "the pin bump is safe"; it cannot and does not replace the real-LLM smoke.

## 3. ADR-022 summary

Full text: `docs/decisions/ADR-022-crewai-version-pin-bump.md`. One-line: **CrewAI optional-dep upper bound goes from `<1.0` to `<2.0`; floor stays at `0.80`; ADR-021 §D1–§D7 untouched.**

Rationale chain:

- ADR-021 §D8 pin was explicitly pre-emptive ("CrewAI hasn't cut a 1.0 yet").
- CrewAI 1.x exists; the environment already has 1.14.3.
- spike13a empirically verifies the event-bus surface is unchanged.
- `<2.0` is SemVer-correct — we get one more major version's worth of runway before having to re-examine.

Consequences documented: no schema change, no CHANGELOG user-facing behavior change, no CLI/API/DB change. It's a manifest + ADR bump.

Three-trigger re-open rule inherits from ADR-021 with CrewAI-version-specific triggers layered on (breaking minor, CrewAI 2.0, observed 0.80–1.14 range failure). One of those fires → revisit the pin.

## 6. Decisions and rationale captured

- **Why a new ADR (-022) instead of editing ADR-021 §D8 in place?** — ADR-021 is Accepted. Accepted ADRs are append-only; revisions need their own ADR that cites the prior by number. This preserves the empirical trail (ADR-021 §D8 was *correct on 2026-04-27*; the pin bump is a *later* empirical finding). Future rounds reading ADR-021 will find ADR-022 listed in cross-references.
- **Why `<2.0` specifically?** — SemVer upper bound. CrewAI 2.0 is a future unknown; we reserve the option to re-examine at that boundary instead of letting unbounded pins silently absorb breakage. `<3.0` would be a stronger bet than the empirical evidence justifies. Exactly one SemVer major of runway.
- **Why not install crewai<1.0 and keep the pin?** — Locks us to a past-EOL line; hostile to downstream users; doesn't solve the underlying problem (the pin's claim that 1.x would break is unverified). The cost of the 30-minute surface probe is lower than the ongoing tax of pinning to a dead line.
- **Why defer R53 P0 one round instead of doing it this slot?** — Budget. Real-LLM spike is 2–3 hr wall clock (first-party CrewAI + real LLM call latency). 30 min pin/probe/ADR fits a single slot cleanly; the combined 3+ hr work risks a third consecutive Option A2 handoff.
- **Why no `spike12` amendment?** — spike12 characterized CrewAI 0.80+ event bus on synthetic events. It's still correct for the 0.80 floor. spike13a is a separate artifact that characterizes 1.14.3. Both live side by side.

## 7. Lessons / Meta

### R53 lesson — "pre-emptive pin" is a smell

ADR-021 §D8's original rationale explicitly said "pre-emptive". A pre-emptive upper bound is a load-bearing guess; guesses get falsified. Future ADRs with `<MAJOR` upper bounds should (a) state the guess explicitly (good: ADR-021 did), (b) include a one-line test script that invalidates the pin the moment the ecosystem moves past it — so the next cron slot *auto-discovers* the falsification. That's a `chronos-spike-authoring` skill enhancement candidate, not in scope this round.

### R53 lesson — spike13a is the template for "pin bump spike"

The probe style (surface check → module path check → handler lifecycle check → scaffold CM check, no real LLM, ~150 lines, ~2 min runtime) is reusable whenever chronos-agent hits the same shape: "our pin says this upstream version shouldn't work, but the ecosystem says it does". Capture in `chronos-spike-authoring` skill if the pattern recurs.

### R53 lesson — the R52 Option A2 handoff was structurally load-bearing

R52's Option A2 close-out meant R53 started from a clean, green baseline. If R52 had shipped the scaffold with the pin bug embedded (e.g. because the cron-slot-handoff-recovery checklist didn't include "try actually importing the target extra under the current env"), R53 would have spent half its budget debugging a Heisenbug instead of writing ADR-022 cleanly. Upgrade candidate for `cron-slot-handoff-recovery` skill: add "if the skill adds a new optional dep, actually try `.venv/bin/python -c 'import <dep>; print(<dep>.__version__)'` before marking Option A2 complete". Noted for the next time the skill is touched.
